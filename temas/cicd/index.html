CI/CD: Integración y Despliegue Continuo

En resumen: CI es automáticamente testear y construir tu código cada vez que alguien hace un cambio. CD es automáticamente deployear ese código cuando todo funciona. El objetivo: deployear con confianza, rápido y seguido.

1. El problema que resuelven

Imaginá un equipo de 5 desarrolladores trabajando en la misma aplicación. Cada uno escribe código en su branch. Si no hay CI/CD, el día del deploy tenés que:

  • Juntar todos los cambios manualmente
  • Resolver conflictos de merge
  • Correr tests (si los tenés)
  • Construir el proyecto
  • Subirlo al servidor
  • Cruzar los dedos para que no se rompa nada

CI/CD automatiza todo esto. Cada vez que alguien hace push, el sistema automáticamente testea, construye y deployea.

2. El pipeline típico

Push

Código al repo

→

Build

Compilar/empaquetar

→

Test

Tests automáticos

→

Deploy

Publicar en producción

Si algo falla en cualquier paso, el pipeline se detiene y notifica al equipo. Nada se publica si no funciona.

3. CI: Integración Continua

CI es la práctica de integrar el código de todos los desarrolladores frecuentemente. En vez de esperar semanas para juntar todo, cada push se integra y verifica automáticamente.


// .github/workflows/ci.yml (GitHub Actions)
name: CI
on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm install
      - run: npm test
      - run: npm run build

Lo que CI verifica en cada push:

  • El código compila sin errores
  • Los tests pasan
  • El lint no reporta problemas
  • La cobertura de tests no bajó

4. CD: Despliegue Continuo vs Entrega Continua

Hay dos variantes de CD:

Continuous Delivery (Entrega Continua): El código está siempre listo para deployear, pero alguien aprueba manualmente cuándo hacerlo.

Continuous Deployment (Despliegue Continuo): Cada cambio que pasa los tests se deployea automáticamente a producción. Sin intervención humana.

Delivery

Automático hasta staging
Aprobación manual para prod

vs

Deployment

Todo automático
Sin aprobación manual

6. Buenas prácticas

  • El pipeline debe ser rápido: Si tarda 30 minutos, los desarrolladores no lo van a esperar. Objetivo: menos de 10 minutos.
  • Fallá rápido: Poné los checks más baratos primero (lint, tests unitarios) antes de los costosos (build, deploy).
  • No saltés tests: Si el pipeline falla, arreglá el código. No deshabilites el test para que pase.
  • Versioná tu pipeline: El archivo de CI/CD debe estar en el repo, no configurado manualmente en la plataforma.
  • Usá cache: Cacheá dependencias y artefactos entre runs para acelerar el pipeline.
  • Variables de entorno: Secrets y configuración van en variables de entorno, nunca en el código.

7. Estrategias de deploy

  • Rolling update: Reemplazás instancias viejas por nuevas gradualmente. Sin downtime.
  • Blue-green: Tenés dos ambientes idénticos. Deployás al nuevo y redirigís el tráfico de golpe. Si algo falla, volvés al anterior.
  • Canary: Deployás a un porcentaje pequeño de usuarios primero. Si no hay errores, expandís gradualmente.
  • Feature flags: El código está desplegado pero desactivado. Lo activás con un switch sin necesidad de redeployear.

La meta: Deployear debería ser aburrido. Si deployear te da ansiedad, tu pipeline necesita mejorar. El CI/CD existe para que puedas hacer cambios con confianza.
