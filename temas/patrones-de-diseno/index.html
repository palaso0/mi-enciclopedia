1. ¿Qué son los patrones de diseño?

Imagina que eres carpintero. No inventas la silla cada vez que alguien te pide una: ya sabes que para hacer una silla necesitas cuatro patas, un asiento y un respaldo. El patrón es la receta que funciona siempre, pero que puedes adaptar a cada cliente.

Los patrones de diseño son exactamente eso: soluciones reutilizables a problemas comunes que aparecen una y otra vez al diseñar software. No son código copiar-pegar, sino plantillas conceptuales que te ayudan a tomar mejores decisiones de arquitectura.

El libro original Design Patterns (1994) de Erich Gamma, Richard Helm, Ralph Johnson y John Vlissides — el "Gang of Four" — documentó 23 patrones que siguen siendo la base del diseño de software moderno.

2. Patrones Creacionales

Resuelven el problema de cómo crear objetos de manera flexible y reutilizable. En lugar de hacer new Clase() por todos lados, encapsulan la lógica de creación.

Creacional Singleton

💡 Es como el director de una escuela: solo puede haber uno. Si necesitas hablar con "el director", siempre es la misma persona, sin importar quién lo pida.
Problema: Necesitas exactamente una instancia de una clase en toda la aplicación (una conexión a base de datos, un logger, un configurador).
Solución: La clase se crea a sí misma y guarda la instancia. Un método estático la devuelve siempre la misma. 

class Singleton:
    _instancia = None
    
    def __new__(cls):
        if cls._instancia is None:
            cls._instancia = super().__new__(cls)
        return cls._instancia
    
    def configurar(self, db_url):
        self.db_url = db_url

# Siempre es la misma instancia
config1 = Singleton()
config2 = Singleton()
print(config1 is config2)  # True

Creacional Factory Method

💡 Es como una fábrica de muebles: cada sucursal (subclase) decide qué tipo de mueble crear, pero el proceso de pedido (método) es el mismo.
Problema: Quieres crear objetos pero no sabes exactamente qué clase necesitas hasta runtime.
Solución: Delega la creación a subclases que implementan un método fábrica. 

from abc import ABC, abstractmethod

class Notificacion(ABC):
    @abstractmethod
    def enviar(self, mensaje):
        pass

class NotificacionEmail(Notificacion):
    def enviar(self, mensaje):
        return f"Email: {mensaje}"

class NotificacionSMS(Notificacion):
    def enviar(self, mensaje):
        return f"SMS: {mensaje}"

class FabricaNotificacion(ABC):
    @abstractmethod
    def crear_notificacion(self) -> Notificacion:
        pass

class FabricaEmail(FabricaNotificacion):
    def crear_notificacion(self):
        return NotificacionEmail()

# Uso
fabrica = FabricaEmail()
notif = fabrica.crear_notificacion()
notif.enviar("Hola")  # "Email: Hola"

Creacional Abstract Factory

💡 Es como un local de IKEA: entrás a la sección "Cocina" y todo combina (olla, sartén, cuchillos del mismo estilo). La fábrica abstracta crea familias enteras de productos compatibles.
Problema: Necesitas crear grupos de objetos relacionados sin especificar sus clases concretas.
Solución: Una interfaz define cómo crear cada tipo de producto, y cada implementación concreta crea una familia consistente. 

class FactoryUI(ABC):
    @abstractmethod
    def crear_boton(self): pass
    @abstractmethod
    def crear_input(self): pass

class FactoryMaterial(FactoryUI):
    def crear_boton(self):
        return BotonMaterial()
    def crear_input(self):
        return InputMaterial()

# La app no sabe qué fábrica usa
def armar_formulario(factory: FactoryUI):
    boton = factory.crear_boton()
    input_ = factory.crear_input()
    return boton, input_

Creacional Builder

💡 Es como armar un combo de hamburguesa: primero la carne, después el queso, luego la lechuga... puedes personalizar paso a paso sin enredarte con 20 parámetros en el constructor.
Problema: Un objeto tiene muchos atributos opcionales y el constructor se vuelve un desastre con 15 parámetros.
Solución: Construí el objeto paso a paso con una interfaz fluida (encadenar métodos). 

class QueryBuilder:
    def __init__(self):
        self._tabla = None
        self._condiciones = []
        self._orden = None
        self._limite = None
    
    def tabla(self, nombre):
        self._tabla = nombre
        return self  # permite encadenar
    
    def donde(self, condicion):
        self._condiciones.append(condicion)
        return self
    
    def construir(self):
        sql = f"SELECT * FROM {self._tabla}"
        if self._condiciones:
            sql += " WHERE " + " AND ".join(self._condiciones)
        return sql

# Uso fluido
query = (QueryBuilder()
    .tabla("usuarios")
    .donde("activo = true")
    .donde("edad > 18")
    .construir())

Creacional Prototype

💡 Es como un molde de arcilla: en lugar de crear cada escultura desde cero, copias un molde existente y lo modificas. Es más rápido que empezar de cero.
Problema: Crear objetos nuevos es costoso (conexiones, carga de datos) y quieres reutilizar existentes.
Solución: Clona objetos existentes en lugar de crear nuevos desde cero. 

import copy

class Prototype:
    def clone(self):
        return copy.deepcopy(self)

class RobotPrototipo(Prototype):
    def __init__(self, nombre, armas, color):
        self.nombre = nombre
        self.armas = armas
        self.color = color

# Original caro de crear
robot_base = RobotPrototipo("RX-78", ["beam", "escudo"], "blanco")

# Clonar es rápido
robot_rojo = robot_base.clone()
robot_rojo.nombre = "RX-78-R"
robot_rojo.color = "rojo"

3. Patrones Estructurales

Resuelven cómo componer objetos para formar estructuras más grandes y flexibles. Se enfocan en simplificar relaciones entre componentes.

Estructural Adapter

💡 Es como un adaptador de enchufe: tienes un enchufe americano y una ficha europea. El adaptador los conecta sin cambiar ninguno de los dos.
Problema: Tienes una clase cuya interfaz no es compatible con la que necesitas usar.
Solución: Envuelve la clase existente en una que traduzca la interfaz. 

class ViejoServicio:
    def enviar_xml(self, datos):
        return f"{datos}"

# El adapter convierte la interfaz
class AdapterServicio:
    def __init__(self, servicio_viejo):
        self.servicio = servicio_viejo
    
    def enviar_json(self, datos):
        xml = self.servicio.enviar_xml(datos)
        return f'{{"raw": "{xml}"}}'

Estructural Bridge

💡 Es como un control remoto universal: el control funciona igual sin importar si controla un TV Sony, Samsung o LG. Podés cambiar la televisión sin cambiar el control.
Problema: Tienes dos dimensiones de variación y no quieres crear una clase para cada combinación.
Solución: Separa la abstracción de la implementación en dos jerarquías independientes. 

from abc import ABC, abstractmethod

class FormaDraw(ABC):
    @abstractmethod
    def dibujar(self, x, y): pass

class CirculoDraw(FormaDraw):
    def dibujar(self, x, y):
        return f"Círculo en ({x},{y})"

class Forma:
    def __init__(self, draw: FormaDraw):
        self.draw = draw
    
    def render(self, x, y):
        return self.draw.dibujar(x, y)

Estructural Composite

💡 Es como una caja de herramientas: adentro puede haber un martillo (hoja) o otra caja con más herramientas (compuesto). Tratás todo igual.
Problema: Tienes objetos individuales y composiciones de objetos, y quieres tratarlos de forma uniforme.
Solución: Compón objetos en estructuras de árbol y trata hojas y ramas con la misma interfaz. 

class Componente(ABC):
    @abstractmethod
    def precio(self): pass

class Producto(Componente):
    def __init__(self, nombre, precio_):
        self.nombre = nombre
        self._precio = precio_
    
    def precio(self):
        return self._precio

class Caja(Componente):
    def __init__(self, nombre):
        self.nombre = nombre
        self.items = []
    
    def agregar(self, item):
        self.items.append(item)
        return self
    
    def precio(self):
        return sum(item.precio() for item in self.items)

Estructural Decorator

💡 Es como vestirte: primero la remera (objeto base), después un saco (decorador), después una bufanda (otro decorador). Cada capa agrega funcionalidad sin cambiar la remera.
Problema: Quieres agregar responsabilidades a objetos de forma dinámica, sin crear subclases para cada combinación.
Solución: Envuelve el objeto en otros objetos que agregan comportamiento. 

class Cafetera:
    def costo(self):
        return 50

class DecoradorCafetera:
    def __init__(self, cafetera):
        self.cafetera = cafetera
    
    def costo(self):
        return self.cafetera.costo()

class Leche(DecoradorCafetera):
    def costo(self):
        return self.cafetera.costo() + 20

# Combinar libremente
cafe = Cafetera()
cafe_special = Crema(Leche(cafe))
print(f"${cafe_special.costo()}")  # $100

Estructural Facade

💡 Es como un recepcionista de hotel: en lugar de hablar con el chef, el limpieza, el conserje y el portero, le dices todo al recepcionista y él coordina todo por ti.
Problema: Tienes un subsistema complejo con muchas clases y quieres simplificar la interfaz para el cliente.
Solución: Creá una clase simple que delegue las llamadas al subsistema complejo. 

class Computadora:
    def __init__(self):
        self.cpu = CPU()
        self.mem = Memoria()
        self.disco = Disco()
    
    def encender(self):
        pasos = []
        pasos.append(self.cpu.freeze())
        pasos.append(self.mem.load("0x00", "bootloader"))
        pasos.append(self.cpu.execute())
        return pasos

Estructural Flyweight

💡 Es como un diccionario: en lugar de crear una definición nueva cada vez que alguien dice "gato", reutilizás la misma definición que ya existe.
Problema: Tienes millones de objetos similares y se te acaba la memoria por duplicar datos idénticos.
Solución: Comparte datos inmutables entre muchos objetos pequeños. 

class FlyweightArbol:
    _pool = {}
    
    def __new__(cls, tipo, color):
        key = (tipo, color)
        if key not in cls._pool:
            obj = super().__new__(cls)
            obj.tipo = tipo
            obj.color = color
            cls._pool[key] = obj
        return cls._pool[key]

# Solo 3 objetos reales en memoria
arboles = []
for i in range(10000):
    arboles.append(FlyweightArbol("pino", "verde"))

Estructural Proxy

💡 Es como un asistente personal: en lugar de que el CEO atienda cada llamada, el asistente filtra, valida y solo pasa lo importante.
Problema: Quieres controlar acceso a un objeto (por seguridad, rendimiento o pereza).
Solución: Pon un objeto intermedio que controle el acceso al real. 

class ProxyServicio:
    def __init__(self):
        self._servicio = None
        self._cache = {}
    
    def consultar(self, dni):
        if dni in self._cache:
            return self._cache[dni]
        
        if self._servicio is None:
            self._servicio = ServicioReal()
        
        resultado = self._servicio.consultar(dni)
        self._cache[dni] = resultado
        return resultado

4. Patrones de Comportamiento

Resuelven cómo los objetos interactúan y distribuyen responsabilidades. Se enfocan en la comunicación y el flujo de control.

Comportamiento Chain of Responsibility

💡 Es como una cadena de mando en el ejército: un soldado no puede resolver un problema, se lo pasa al sargento; si tampoco puede, al teniente; y así hasta llegar al general.
Problema: Varios objetos pueden manejar una petición, pero no sabes cuál al principio.
Solución: Pasa la petición por una cadena de manejadores hasta que uno la resuelva. 

class Manejador:
    def __init__(self, siguiente=None):
        self.siguiente = siguiente
    
    def manejar(self, monto):
        if self.siguiente:
            return self.siguiente.manejar(monto)
        return None

class Supervisor(Manejador):
    def __init__(self, limite, siguiente=None):
        super().__init__(siguiente)
        self.limite = limite
    
    def manejar(self, monto):
        if monto <= self.limite:
            return f"Supervisor aprobó ${monto}"
        return super().manejar(monto)

Comportamiento Command

💡 Es como un ticket de restaurante: el mesero escribe tu pedido en un papelito. La cocina recibe el papelito y lo ejecuta. Podés guardar el papelito, deshacerlo, o pasarlo a otro cocinero.
Problema: Quieres desacoplar quién ejecuta una acción de quién la pidió, o necesitas deshacer/rehacer operaciones.
Solución: Encapsula cada acción como un objeto con ejecutar() y deshacer(). 

class Comando(ABC):
    @abstractmethod
    def ejecutar(self): pass
    @abstractmethod
    def deshacer(self): pass

class Escribir(Comando):
    def __init__(self, documento, texto):
        self.doc = documento
        self.texto = texto
    
    def ejecutar(self):
        self.doc.contenido += self.texto

Comportamiento Observer

💡 Es como suscribirte a un canal de YouTube: cuando el canal sube un video, todos los suscriptores reciben una notificación automática.
Problema: Un objeto cambia y varios otros necesitan enterarse, pero no quieres acoplarlos.
Solución: Los objetos se suscriben a notificaciones del sujeto y se actualizan cuando algo cambia. 

class Sujeto:
    def __init__(self):
        self._observadores = []
        self._estado = None
    
    def suscribir(self, observador):
        self._observadores.append(observador)
    
    def notificar(self):
        for obs in self._observadores:
            obs.actualizar(self._estado)
    
    @estado.setter
    def estado(self, valor):
        self._estado = valor
        self.notificar()

Comportamiento Strategy

💡 Es como elegir camino al trabajo: puedes ir en auto, en colectivo o caminando. El destino es el mismo, pero la estrategia cambia según el contexto.
Problema: Tienes múltiples algoritmos para resolver un problema y quieres intercambiarlos en runtime.
Solución: Define cada algoritmo como una estrategia intercambiable. 

class EstrategiaDescuento(ABC):
    @abstractmethod
    def calcular(self, total): pass

class DescuentoNormal(EstrategiaDescuento):
    def calcular(self, total):
        return total

class DescuentoVIP(EstrategiaDescuento):
    def calcular(self, total):
        return total * 0.8

Comportamiento Iterator

💡 Es como recorrer un álbum de fotos: no necesitas sacar todas las fotos a la vez. Vas pasando una por una con la mano (el iterador).
Problema: Quieres recorrer una colección sin exponer su estructura interna.
Solución: Crea un objeto que recorra la colección paso a paso. 

class IteradorArbol:
    def __init__(self, raiz):
        self._pila = [raiz]
    
    def __next__(self):
        if not self._pila:
            raise StopIteration
        nodo = self._pila.pop()
        for hijo in reversed(nodo.hijos):
            self._pila.append(hijo)
        return nodo.valor

5. ¿Cuándo usar cada uno?

No todos los patrones son útiles en todos los proyectos. Acá va una guía rápida:

Usa un patrón cuando...

✓ Tienes un problema recurrente que ya tiene solución documentada
✓ El patrón reduce la complejidad en lugar de agregarla
• Tu equipo conoce el patrón o está dispuesto a aprenderlo
• El beneficio justifica la abstracción adicional

NO uses un patrón cuando...

✗ Es over-engineering para el problema actual
✗ Nadie del equipo lo entenderá
✗ Agrega complejidad sin beneficio claro
✗ "Porque sí" no es una razón válida

Regla de oro: Primero escribe el código más simple que funcione. Si luego descubres que necesitas un patrón, refactoriza. No empieces con patrones.
