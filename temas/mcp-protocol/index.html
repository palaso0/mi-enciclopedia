MCP: Model Context Protocol

Imagina que cada dispositivo electrónico tuviera su propio tipo de puerto de carga: uno para cables redondos, otro para cables planos, otro magnético. Sería un caos. USB unificó todo eso: un solo puerto que funciona con cualquier dispositivo. El Model Context Protocol (MCP), creado por Anthropic, hace lo mismo para los agentes de IA: un estándar universal que permite conectar cualquier modelo con cualquier fuente de datos o herramienta, sin importar quién hizo cada uno.

La analogía USB: Antes de MCP, cada integración entre un LLM y una herramienta era custom: si querías que tu modelo accediera a Notion, escribías código específico para Notion. Si luego querías Slack, escribías otro código diferente. MCP estandariza esa conexión: el proveedor de datos implementa el protocolo una vez, y cualquier modelo compatible puede conectarse automáticamente. Es un "puerto universal" para la IA.

1. ¿Qué es MCP exactamente?

MCP es un protocolo de comunicación abiño que define cómo los modelos de IA (clientes) se conectan con fuentes de datos y herramientas externas (servidores). No es una API, no es un SDK, no es un framework: es un protocolo, como HTTP lo es para la web. Cualquier plataforma puede implementarlo.

Estándar abierto

Cualquier desarrollador puede implementar un servidor MCP o un cliente MCP. No hay vendor lock-in ni licencias restrictivas.

Transporte flexible

Funciona sobre stdio (para procesos locales), HTTP/SSE (para servicios remotos) y potencialmente WebSocket. Se adapta al entorno de ejecución.

Descubrimiento automático

El cliente puede preguntar al servidor qué herramientas, recursos y prompts tiene disponibles, sin configuración previa.

Seguridad integrada

El protocolo define mecanismos de autenticación y control de acceso para proteger los datos del servidor.

2. Arquitectura cliente-servidor

MCP utiliza una arquitectura cliente-servidor donde el cliente es la aplicación de IA (como Claude Desktop o un agente custom) y el servidor es la fuente de datos o herramienta que se conecta:

Cliente MCP

La aplicación de IA que necesita acceder a datos y herramientas. Ej: Claude Desktop, VS Code con Copilot, un agente personalizado.

⇄

Servidor MCP

El proveedor de datos o herramientas. Ej: un servidor de Notion, de GitHub, de una base de datos, de un sistema de archivos.

La comunicación entre cliente y servidor se realiza mediante mensajes JSON-RPC 2.0. El cliente envía solicitudes y el servidor responde con resultados. También puede enviar notificaciones unidireccionales para actualizaciones en tiempo real.

3. Los tres pilares: Recursos, Herramientas y Prompts

MCP define tres tipos de primitives que un servidor puede exponer. Cada una tiene un propósito distinto y se usa de manera diferente:

Recursos (Resources)

Datos que el servidor pone a disposición del modelo. Son solo lectura. Ej: el contenido de un archivo, datos de una API, registros de una base de datos. El modelo puede leerlos pero no modificarlos directamente.

Herramientas (Tools)

Funciones que el modelo puede invocar para ejecutar acciones. Son ejecutables. Ej: crear un archivo, enviar un mensaje, ejecutar una consulta SQL. Requieren aprobación del usuario en muchos casos.

Prompts

Plantillas predefinidas que el servidor puede ofrecer al modelo. Ayudan a estructurar interacciones comunes. Ej: un prompt para revisar código, otro para resumir documentos, otro para generar tests.

La diferencia clave: Las recursos son datos pasivos (como abrir un libro). Las herramientas son acciones activas (como usar un destornillador). Los prompts son plantillas de interacción (como un formulario pre-armado). Un servidor MCP puede exponer las tres cosas o solo una, dependiendo de su diseño.

4. ¿Por qué importa la estandarización?

Sin un protocolo estándar, cada integración entre LLMs y herramientas es un proyecto custom. Esto crea un problema de escalabilidad:

Sin MCP (custom) Con MCP (estándar)
N proveedores × M modelos = N×M integraciones N proveedores + M modelos = N+M implementaciones
Cada integración es única y frágil Una implementación funciona con todos los clientes
Actualizar una API rompe la integración El protocolo gestiona compatibilidad
No hay descubrimiento automático El cliente descubre herramientas al conectarse
Cada desarrollador resuelve autenticación a su manera Mecanismos de auth estandarizados

La estandarización también beneficia a los desarrolladores de herramientas: implementan el protocolo una vez y su herramienta funciona con Claude, GPT, Gemini, y cualquier otro modelo compatible. No necesitan escribir integración separada para cada proveedor de IA.

5. MCP vs. Function Calling

MCP y function calling resuelven problemas similares pero en niveles diferentes. No son competidores; son complementarios:

Aspecto Function Calling MCP
Alcance Llamadas a funciones específicas definidas por el desarrollador Protocolo completo para conectar fuentes de datos y herramientas
Descubrimiento Las funciones se definen estáticamente en el código El cliente descubre automáticamente qué está disponible
Portabilidad Cada proveedor tiene su propio formato Protocolo universal compatible con múltiples proveedores
Recursos Solo ejecuta funciones También accede a datos (recursos) y ofrece prompts
Transporte HTTP directo o SDK del proveedor stdio, HTTP/SSE,WebSocket con handshake definido

En la práctica, un agente puede usar MCP para conectarse a un servidor que, internamente, expone sus capacidades como tools. MCP actúa como la capa de transport y descubrimiento, mientras que las tools individuales son las unidades de ejecución.

6. Implementaciones reales y el ecosistema

Desde su lanzamiento a finales de 2024, MCP ha generado un ecosistema creciente de implementaciones:

Claude Desktop

Fue la primera aplicación en soportar MCP. Permite conectar servidores locales y remotos para que Claude acceda a archivos, bases de datos y APIs.

Servidores comunitarios

Existen servidores MCP para: GitHub, Notion, Slack, PostgreSQL, MySQL, Slack, Google Drive, Filesystem, Puppeteer, y muchos más.

SDK oficial

Anthropic provee SDKs en TypeScript y Python para construir servidores y clientes MCP. Simplifican la implementación del protocolo.

Adopción

OpenAI, Google y otros proveedores han anunciado compatibilidad o interés en MCP, convirtiéndolo en un estándar de facto emergente.

7. Simulación: Un servidor MCP en acción

Veamos cómo se ve la interacción entre un cliente MCP y un servidor que expone acceso a una base de datos:

import json

class ServidorMCP:
    """Simulación de un servidor MCP que expone una base de datos"""
    
    def __init__(self):
        self.herramientas = {
            "consultar_usuarios": {
                "descripcion": "Busca usuarios por nombre o email",
                "parametros": {
                    "tipo": "object",
                    "propiedades": {
                        "nombre": {"tipo": "string"},
                        "email": {"tipo": "string"}
                    }
                }
            },
            "crear_pedido": {
                "descripcion": "Crea un nuevo pedido para un usuario",
                "parametros": {
                    "tipo": "object",
                    "propiedades": {
                        "usuario_id": {"tipo": "integer"},
                        "productos": {"tipo": "array"},
                        "direccion": {"tipo": "string"}
                    },
                    "requeridos": ["usuario_id", "productos"]
                }
            }
        }
        
        self.recursos = {
            "file:///docs/catalogo.json": {
                "nombre": "Catálogo de productos",
                "tipo": "application/json"
            },
            "file:///docs/politicas.md": {
                "nombre": "Políticas de envío",
                "tipo": "text/markdown"
            }
        }
    
    def manejar_solicitud(self, solicitud):
        """Procesa mensajes JSON-RPC 2.0 del cliente"""
        metodo = solicitud.get("method")
        params = solicitud.get("params", {})
        
        if metodo == "tools/list":
            # El cliente pregunta: ¿qué herramientas tenés?
            return {
                "tools": [
                    {"name": k, "description": v["descripcion"],
                     "inputSchema": v["parametros"]}
                    for k, v in self.herramientas.items()
                ]
            }
        
        elif metodo == "tools/call":
            # El cliente pide ejecutar una herramienta
            nombre = params.get("name")
            args = params.get("arguments", {})
            return self.ejecutar_herramienta(nombre, args)
        
        elif metodo == "resources/list":
            # El cliente pregunta: ¿qué recursos tenés?
            return {
                "resources": [
                    {"uri": uri, "name": r["nombre"], "mimeType": r["tipo"]}
                    for uri, r in self.recursos.items()
                ]
            }
        
        elif metodo == "resources/read":
            # El cliente quiere leer un recurso específico
            uri = params.get("uri")
            return self.leer_recurso(uri)

# Flujo de la conversación cliente ↔ servidor
print("1. Cliente se conecta al servidor MCP")
print("2. Cliente envía: tools/list → Descubre las herramientas disponibles")
print("3. Cliente envía: resources/list → Descubre los recursos disponibles")
print("4. LLM decide usar 'consultar_usuarios' con nombre='Pablo'")
print("5. Cliente envía: tools/call con los argumentos")
print("6. Servidor ejecuta la consulta y retorna resultados")
print("7. LLM integra los resultados en una respuesta natural")

El futuro de MCP: A medida que más proveedores adopten el protocolo, los agentes de IA podrán conectarse a cualquier servicio con un estándar统一. Esto reduce drásticamente la fricción para crear agentes verdaderamente útiles que puedan interactuar con el mundo real sin necesidad de integraciones custom para cada servicio.
