Complejidad Temporal y Espacial (Big O)

En resumen: Big O es una forma de medir qué tan rápido crece el tiempo de ejecución o el consumo de memoria de un algoritmo a medida que crecen los datos de entrada. No te dice cuántos segundos tarda, sino cómo se comporta cuando le das más trabajo.

1. Por qué importa

Imaginá que tenés una lista de 100 números y querés encontrar el mayor. Podés recorrer toda la lista: eso son 100 operaciones. Si tuvieras 1.000 números, serían 1.000 operaciones. Si tuvieras 1.000.000, serían un millón.

Pero ¿qué pasa si tu algoritmo hace una comparación por cada par de números? Con 100 elementos serían 10.000 comparaciones. Con 1.000 elementos, un millón. Con un millón, un billón. El tiempo crece mucho más rápido que los datos.

Big O te permite predecir este comportamiento sin importar la computadora, el lenguaje o los datos específicos. Es una comparación pura de cómo escala.

2. Notación Big O

O(1)

Constante

Siempre tarda lo mismo, sin importar el tamaño de entrada. El mejor caso posible.

O(log n)

Logarítmica

Cada paso reduce el problema a la mitad. Muy eficiente para datos grandes.

O(n)

Lineal

Crece proporcionalmente con los datos. Si duplicás los datos, duplicás el tiempo.

O(n²)

Cuadrática

Por cada elemento, recorre todos los demás. Se vuelve lento rápido.

O(2ⁿ)

Exponencial

Duplica el trabajo por cada elemento nuevo. Solo viable para entradas muy pequeñas.

3. Ejemplos prácticos con gráficos

O(1) — Acceso por índice


const arr = [10, 20, 30, 40, 50];
const valor = arr[3]; // Siempre 1 operación

O(n) — Buscar en una lista


function buscar(lista, objetivo) {
  for (let i = 0; i < lista.length; i++) {
    if (lista[i] === objetivo) return i;
  }
  return -1;
}

O(n²) — Comparar todos contra todos


// Bubble sort: dos loops anidados
function burbuja(arr) {
  for (let i = 0; i < arr.length; i++) {
    for (let j = 0; j < arr.length - 1; j++) {
      if (arr[j] > arr[j + 1]) {
        [arr[j], arr[j + 1]] = [arr[j + 1], arr[j]];
      }
    }
  }
}

O(log n) — Búsqueda binaria


function busquedaBinaria(arr, objetivo) {
  let inicio = 0, fin = arr.length - 1;
  while (inicio <= fin) {
    const medio = Math.floor((inicio + fin) / 2);
    if (arr[medio] === objetivo) return medio;
    if (arr[medio] < objetivo) inicio = medio + 1;
    else fin = medio - 1;
  }
  return -1;
}

5. Cómo optimizar: el poder de cambiar la complejidad

La mejora más grande que podés hacer no es optimizar el código existente, sino cambiar el algoritmo. Un algoritmo O(n²) optimizado sigue siendo O(n²). Pero si lo cambias a O(n) o O(log n), el cambio es exponencial.

Ejemplo: buscar un elemento en una lista ordenada

MAL: Búsqueda lineal — O(n)


function buscarLineal(arr, obj) {
  for (let i = 0; i < arr.length; i++) {
    if (arr[i] === obj) return i;
  }
  return -1;
}
// 1,000,000 de elementos
// = hasta 1,000,000 comparaciones

BIEN: Búsqueda binaria — O(log n)


function buscarBinaria(arr, obj) {
  let ini = 0, fin = arr.length - 1;
  while (ini <= fin) {
    const mid = (ini + fin) >> 1;
    if (arr[mid] === obj) return mid;
    if (arr[mid] < obj) ini = mid + 1;
    else fin = mid - 1;
  }
  return -1;
}
// 1,000,000 de elementos
// = solo 20 comparaciones

Ejemplo: encontrar duplicados

MAL: Dos loops — O(n²)


function tieneDuplicados(arr) {
  for (let i = 0; i < arr.length; i++) {
    for (let j = i + 1; j < arr.length; j++) {
      if (arr[i] === arr[j]) return true;
    }
  }
  return false;
}
// 10,000 elementos = 50,000,000 comparaciones

BIEN: Set + un loop — O(n)


function tieneDuplicados(arr) {
  const vistos = new Set();
  for (const item of arr) {
    if (vistos.has(item)) return true;
    vistos.add(item);
  }
  return false;
}
// 10,000 elementos = 10,000 operaciones

Ejemplo: contar frecuencia de caracteres

MAL: Buscar cada vez — O(n²)


function contarFrecuencia(str) {
  const chars = [...new Set(str)];
  return chars.map(c => ({
    char: c,
    count: str.split(c).length - 1
  }));
}
// split() recorre el string completo
// por cada carácter único

BIEN: Un solo recorrido — O(n)


function contarFrecuencia(str) {
  const freq = {};
  for (const c of str) {
    freq[c] = (freq[c] || 0) + 1;
  }
  return freq;
}
// Un solo loop = n operaciones

5. Complejidad espacial

No solo importa el tiempo: también cuánta memoria usa tu algoritmo. Un algoritmo puede ser rápido pero gastar mucha RAM.


// O(1) espacial: solo usa variables extra fijas
function maximo(arr) {
  let mejor = arr[0];
  for (let i = 1; i < arr.length; i++) {
    if (arr[i] > mejor) mejor = arr[i];
  }
  return mejor;
}

// O(n) espacial: crea un nuevo array del mismo tamaño
function duplicar(arr) {
  return arr.map(x => x * 2);
}

6. Reglas para analizar complejidad

  • Ignorá las constantes: O(2n) es lo mismo que O(n). Lo que importa es la tendencia, no el factor exacto.
  • Ignorá los términos menores: O(n² + n) se simplifica a O(n²) porque el cuadrático domina.
  • Lo diferente es lo que importa: Si un algoritmo tiene dos partes, una O(n) y otra O(n²), la complejidad total es O(n²).
  • Mejor, peor y promedio: Big O normalmente describe el peor caso. También existen Omega (mejor caso) y Theta (promedio).

La regla práctica: Si tu algoritmo tiene un solo loop, es O(n). Si tiene dos loops anidados, es O(n²). Si dividís el problema a la mitad en cada paso, es O(log n). Si accedés directamente por índice, es O(1).
