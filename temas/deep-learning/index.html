1. Transformers y el mecanismo de atención

Si alguna vez usaste ChatGPT, usaste un Transformer. Esta arquitectura, creada en 2017 por Google, cambió por completo cómo las computadoras entienden nuestro lenguaje. En lugar de procesar la información palabra por palabra, aprendieron a mirar todo el contexto a la vez.

1.1 La diferencia clave: Leer con lupa vs. Mirar la foto completa

Para entender por qué son especiales, comparemos cómo leían los modelos antiguos frente a cómo lee un Transformer:

El modelo antiguo (Redes Recurrentes / RNNs): Imagina que lees un libro con una lupa, viendo solo una palabra a la vez. Para entender la palabra 10, tienes que recordar las 9 anteriores en orden. Si la oración es muy larga, para cuando llegas al final ya te olvidaste de cómo empezó.

El Transformer: En lugar de una lupa, toma una fotografía de toda la página a la vez. No lee en orden secuencial; analiza todas las palabras al mismo tiempo e identifica qué palabras se relacionan entre sí para darles sentido.

1.2 El poder del contexto: El problema de la palabra "banco"

El lenguaje humano está lleno de palabras con múltiples significados (polisemia). Una computadora necesita contexto para saber de qué estamos hablando.

Analicemos estas dos oraciones con el concepto de Atención:

Caso A:

El banco de la plaza estaba roto

→ Aquí, la palabra banco "presta atención" a plaza y roto. El modelo deduce que se refiere a un asiento físico.

Caso B:

El banco central subió las tasas

→ Aquí, la palabra banco "presta atención" a central y tasas. El modelo deduce que se trata de una institución financiera.

1.3 ¿Cómo calcula esto la computadora? (Query, Key y Value)

Para decidir qué palabra se relaciona con cuál, el Transformer usa una fórmula matemática que funciona de manera muy similar a **hacer una búsqueda en YouTube**:

  • Query (Búsqueda): Es lo que la palabra actual está buscando. *(Ejemplo: "Busco palabras que definan mi significado físico o institucional")*.
  • Key (Etiqueta): Es la etiqueta que ofrecen todas las demás palabras de la oración. *(Ejemplo: "plaza" tiene la etiqueta "lugar/exterior", "central" tiene la etiqueta "finanzas")*.
  • Value (Contenido): Si la búsqueda (Query) coincide con una etiqueta (Key), se extrae la información de esa palabra para ajustar el significado final.

Este cruce de búsquedas y etiquetas se realiza mediante multiplicaciones de matrices muy veloces que se pueden procesar en paralelo, lo que permite entrenar modelos con millones de textos de internet.

1.4 Un ejemplo simplificado en código

Para ver cómo se traduce esto a lógica, imagina que calculamos la fuerza de relación (atención) entre las palabras de la oración: "El banco de la plaza".

# Simulación de la matriz de atención para la palabra "banco"
palabras = ["El", "banco", "de", "la", "plaza"]

# Valores calculados por el modelo que miden la relación (de 0 a 1)
atencion_para_banco = {
    "El": 0.05,
    "banco": 1.00,  # Relación consigo mismo
    "de": 0.05,
    "la": 0.05,
    "plaza": 0.85   # ¡Fuerte conexión contextual!
}

# El modelo aprende que "banco" y "plaza" forman un bloque de significado

1.5 El truco final: Positional Encoding

Como el Transformer lee todo al mismo tiempo, no sabe qué palabra va primero y cuál va después. Para él, estas dos oraciones serían idénticas:

  1. "El gato se comió al ratón"
  2. "El ratón se comió al gato"

Para solucionar esto, el Transformer añade una **coordenada de posición** a cada palabra antes de procesarla. Es como darle a cada palabra una etiqueta con su número de orden (Posición 1, Posición 2, etc.), asegurando que entienda la estructura de la oración sin sacrificar la velocidad de lectura paralela.

2. Vision Transformers (ViT)

¿Puede una arquitectura diseñada para procesar texto clasificar imágenes de manera eficiente? En 2020, investigadores de Google demostraron con el paper "An Image is Worth 16x16 Words" que sí es posible. Al tratar una imagen como una secuencia de palabras, los Vision Transformers (ViT) desafiaron el reinado de las Redes Convolucionales (CNNs).

Concepto clave: En lugar de rediseñar el Transformer para visión, ViT adapta la imagen para que encaje exactamente en un Transformer estándar de procesamiento de texto.

2.1 La intuición: Lupa local vs. Rompecabezas global

La diferencia principal entre las redes convolucionales tradicionales y un ViT radica en cómo "miran" la imagen:

  • Red Convolucional (CNN): Es como examinar una foto con una lupa pequeña, buscando patrones locales (bordes, texturas) y combinándolos gradualmente. Sabe por diseño que los píxeles cercanos están relacionados.
  • Vision Transformer (ViT): Corta la foto en piezas de rompecabezas (típicamente de 16x16 píxeles), las aplana y las analiza simultáneamente usando autoatención. No asume por diseño qué pieza está al lado de cuál; aprende la relación espacial analizando los datos.

2.2 El Pipeline: ¿Cómo se convierte una imagen en "palabras"?

Para alimentar a un Transformer, necesitamos pasar de una imagen tridimensional a una secuencia de vectores lineales:

P1
P2
P3
P4
P5
P6
P7
P8
P9

1. Dividimos la imagen en "parches" (Patches) rígidos.

Patch 1 → [v1, v2, v3, ...]
Patch 2 → [v4, v5, v6, ...]
...

2. Aplanamos cada parche en un vector y lo proyectamos al tamaño deseado (Embeddings).

2.3 Los componentes críticos de ViT

Componente Función Por qué es necesario
Patches (16x16) Divide la imagen en fragmentos. Reduce la cantidad de tokens. Analizar píxel por píxel con atención sería computacionalmente imposible (complejidad cuadrática).
Linear Projection Transforma los parches aplanados en vectores (embeddings). Mapea los píxeles a un espacio dimensional que el Transformer entiende.
Positional Embedding Suma coordenadas de posición a cada embedding. Al no tener recurrencia ni convolución, el modelo no sabe en qué parte de la imagen va cada parche si no se lo indicamos.
Class Token [CLS] Un vector extra que se añade al inicio de la secuencia. Este token acumula la información de atención de toda la imagen y se usa al final para hacer la predicción (ej. "gato" o "perro").

2.4 Simulación en código: De imagen a embeddings

Este script de Python demuestra matemáticamente cómo se descompone y proyecta una imagen en parches antes de entrar al Transformer:

# Simulación conceptual de preparación de parches para ViT
import numpy as np

# Configuración básica
H, W, C = 224, 224, 3   # Altura, ancho, canales (RGB) de la imagen
patch_size = 16          # Cada parche será de 16x16 píxeles
d_model = 768            # Dimensión del embedding de salida

# 1. Simular una imagen cargada
imagen = np.random.rand(H, W, C)

# 2. Calcular número de parches
n_patches = (H // patch_size) * (W // patch_size)  # (224/16) * (224/16) = 196 parches
print(f"Número de parches generados: {n_patches}")

# 3. Dividir la imagen conceptualmente y aplanar los parches
# Cada parche tiene un tamaño de: 16 * 16 * 3 = 768 valores
parches_aplanados = np.random.rand(n_patches, patch_size * patch_size * C)
print(f"Dimensiones de los parches aplanados: {parches_aplanados.shape}") # (196, 768)

# 4. Proyección Lineal (matriz de pesos W_p)
W_p = np.random.rand(patch_size * patch_size * C, d_model)
patch_embeddings = np.dot(parches_aplanados, W_p)
print(f"Dimensión final para el Transformer: {patch_embeddings.shape}") # (196, 768)

2.5 El dilema de los datos: Inductive Bias

A diferencia de las CNNs, los Transformers no tienen **sesgo inductivo (inductive bias)** sobre las imágenes. Una CNN "sabe" implícitamente que los píxeles vecinos son importantes y que los patrones se mantienen iguales si se desplazan (invarianza de traslación).

El ViT tiene que aprender todo esto desde cero analizando millones de ejemplos. Por eso:

  • Si se entrena en conjuntos de datos pequeños (como ImageNet-1k), ViT rinde **peor** que una CNN.
  • Si se pre-entrena en conjuntos masivos de datos (como el set de datos privado de Google JFT-300M), ViT **supera con creces** a cualquier CNN tradicional.

3. Flash Attention

Publicada en 2022 por Tri Dao e investigadores de Stanford, Flash Attention no cambia la matemática de la autoatención del Transformer; cambia **cómo se ejecuta en el hardware de la GPU**. Al reestructurar los accesos a la memoria, logró acelerar el entrenamiento hasta 3 veces y permitió que los modelos procesen secuencias de texto masivamente más largas.

El problema real: El cuello de botella en las GPUs modernas no es la velocidad de cálculo (FLOPS), sino la velocidad a la que se transfieren los datos entre la memoria lenta de la GPU y los núcleos rápidos de procesamiento.

3.1 La física del hardware: HBM vs. SRAM

Para entender Flash Attention, debemos entender la GPU como un escritorio de trabajo:

HBM (High Bandwidth Memory) — El Sótano
Matriz Q Matriz K Matriz V Matriz de Atención (Gigante)

Memoria principal de la GPU. Es enorme (80GB+), pero el acceso a ella es muy **lento**.

⇅ Transferencia (Cuello de Botella)
SRAM (Static RAM) — El Escritorio
Núcleo Tensor (Cálculo)

Memoria ultra rápida al lado del procesador. Es muy pequeña (solo unos MBs).

En el mecanismo de atención tradicional, el modelo calcula la matriz de atención de tamaño \(N \times N\) (donde \(N\) es la longitud del texto). Esta matriz es tan gigante que no cabe en la SRAM (el escritorio), por lo que la GPU debe escribirla y leerla constantemente desde la HBM (el sótano) para aplicar operaciónes como Softmax. Esta transferencia es lenta e ineficiente.

3.2 El truco de Flash Attention: "Tiling" (Partición en Bloques)

Flash Attention evita almacenar la gigantesca matriz intermedia de atención. En su lugar, utiliza una técnica llamada **Tiling** (mosaicos):

  • Divide las matrices Query, Key y Value en pequeños bloques que sí caben en la SRAM.
  • Carga un bloque de la HBM a la SRAM.
  • Realiza los cálculos del Softmax de manera parcial sobre ese bloque.
  • Utiliza un truco matemático de re-escalado para unir los resultados Softmax parciales sin perder precisión.
  • Escribe directamente el resultado final en la HBM, evitando escribir la matriz de atención temporal.

3.3 Comparativa de operaciónes en memoria

Operación Atención Tradicional Flash Attention
Complejidad en memoria \(O(N^2)\) (Cuadrática) \(O(N)\) (Lineal respecto a escrituras HBM)
Escrituras en HBM Muy altas (guarda la matriz intermedia) Mínimas (solo lee Q, K, V y escribe la salida)
Límite de Contexto Corto (se queda sin memoria rápidamente) Muy largo (ej: 32k, 128k tokens o más)
Velocidad Lenta por accesos a memoria (Memory-bound) Hasta 3x más rápida (Compute-bound)

3.4 Simulación conceptual: El costo de los accesos a memoria

El siguiente código simula conceptualmente la cantidad de lecturas y escrituras de memoria (I/O) requeridas por ambos algoritmos para procesar una secuencia larga:

# Simulación del costo de acceso a memoria (I/O Complexity)
# N = Longitud de secuencia, d = Dimensión de embedding, M = Tamaño de la SRAM

def costo_memoria_clasico(N, d):
    # Lee Q, K (2Nd) -> Calcula y escribe S (N^2) -> Lee S, V (N^2 + Nd) -> Escribe salida (Nd)
    lecturas = 2 * N * d + N**2 + N * d
    escrituras = N**2 + N * d
    return lecturas + escrituras

def costo_memoria_flash(N, d, M):
    # Divide en bloques de tamaño ~ M/d. Evita escribir la matriz N^2.
    # Costo dominado por leer Q, K, V y re-escribir salidas varias veces
    costo_io = (N**2 * d**2) / M  # Simplificación matemática del paper
    return costo_io

# Pruebas con secuencia de 8000 tokens
N_seq = 8000
d_emb = 64
M_sram = 100000  # Capacidad pequeña en unidades relativas

io_clasico = costo_memoria_clasico(N_seq, d_emb)
io_flash = costo_memoria_flash(N_seq, d_emb, M_sram)

print(f"Accesos a HBM (Tradicional): {io_clasico:,} lecturas/escrituras")
print(f"Accesos a HBM (Flash):       {int(io_flash):,} lecturas/escrituras")
print(f"Reducción de I/O de memoria: ~{io_clasico / io_flash:.1f}x")

3.5 Impacto en los modelos modernos

Gracias a Flash Attention, la industria pudo dar el salto de los 2,048 tokens de contexto originales de GPT-3 a los 128k o 1M de tokens que manejan los modelos actuales. Al no consumir memoria en almacenar la matriz intermedia de atención, las GPUs pueden procesar documentos enteros, libros o códigos fuente masivos de una sola vez.

4. Mixture-of-Experts (MoE)

Modelos como Mixtral 8x7B o el propio GPT-4 no son un solo bloque de redes masivas. Utilizan una arquitectura **Mixture-of-Experts (MoE)**, un diseño en el que se dividen las capas del modelo en múltiples "expertos" especializados, y un enrutador inteligente decide qué expertos deben procesar cada palabra. Esto permite tener modelos gigantescos pero consumiendo una fracción del costo de cómputo.

Dato clave: MoE implementa un tipo de **activación dispersa (sparse)**. A diferencia de un modelo denso tradicional, en el cual se activan todos los parámetros para cada palabra, MoE solo activa un porcentaje pequeño (ej: 10-20%).

4.1 La intuición: La clínica médica

Compara el procesamiento de una consulta en estas dos estructuras:

  • Modelo Denso: Entras a una clínica y para darte un diagnóstico, te atienden **todos** los médicos del hospital (el cardiólogo, el dermatólogo, el pediatra, etc.) al mismo tiempo. Es sumamente ineficiente y caro.
  • Modelo Mixture-of-Experts (MoE): Llegas a recepción (el **Router**). El recepcionista evalúa tu síntoma y te deriva únicamente con el traumatólogo y el radiólogo (los **Expertos**). El resto de médicos siguen descansando.

4.2 El Flujo de Trabajo de un Token en MoE

Token: "código"
↓
Gating Network (Router)
↓ (Top-2 Experts Selected)
Experto 1
(Programación)
Experto 2
(Finanzas)
Experto 3
(Gramática)

Por cada capa MoE de la red:

  1. El token de entrada llega a la red de enrutamiento (**Gating Network**).
  2. El Router calcula una distribución de probabilidad sobre los \(N\) expertos usando una función Softmax.
  3. Se seleccionan los **Top-K expertos** con mayor puntuación (típicamente \(K=1\) o \(K=2\)).
  4. El token se envía y procesa únicamente por los expertos seleccionados.
  5. Las salidas de los expertos se combinan multiplicándose por la probabilidad que les asignó el Router.

4.3 Parámetros Totales vs. Parámetros Activos

Esta es la magia matemática que permite hacer viables los modelos de billones de parámetros:

Modelo Tipo Parámetros Totales Parámetros Activos por Token Costo de Cómputo
Llama 3 70B Denso 70 Billones 70 Billones (100%) Muy Alto
Mixtral 8x7B MoE (Sparse) 47 Billones 13 Billones (~27%) Equivalente a un modelo de 13B

Beneficio: Mixtral tiene la capacidad y el conocimiento de un modelo de 47B de parámetros, pero su velocidad de generación (latencia) y requisitos de cómputo en inferencia son los de un modelo pequeño de 13B.

4.4 Simulación conceptual: El Router MoE

Este fragmento de Python simula la toma de decisiones del Router para asignar un token a los mejores 2 expertos de un pool de 4:

# Simulación del Router (Gating Network) en una capa MoE
import numpy as np

def softmax(x):
    e_x = np.exp(x - np.max(x))
    return e_x / e_x.sum()

# 1. Definir nombres de expertos en nuestro pool
expertos = ["Experto Matemático", "Experto Programación", "Experto Creativo", "Experto Traductor"]

# 2. Simular un token de entrada (vector) y los pesos del Router
# Digamos que el token tiene que ver con escribir código de Python
vector_token = np.array([0.1, 0.9, 0.2, 0.4]) 
pesos_router = np.random.rand(4, 4)  # Pesos entrenados del Router

# 3. Calcular afinidad del token con cada experto
logits = np.dot(vector_token, pesos_router)
probabilidades = softmax(logits)

# 4. Seleccionar los mejores K=2 expertos (Top-2 Routing)
top_k_indices = np.argsort(probabilidades)[::-1][:2]

print("Probabilidades asignadas por el Router:")
for idx, prob in enumerate(probabilidades):
    print(f" - {expertos[idx]}: {prob:.2%}")

print("\nExpertos ACTIVADOS para este token:")
for idx in top_k_indices:
    print(f" -> {expertos[idx]} (Prob: {probabilidades[idx]:.2%})")

4.5 Los desafíos de MoE

  • Memoria RAM / VRAM: Aunque solo actives 13B de parámetros por token, **tienes que cargar los 47B de parámetros en la memoria de la tarjeta de video**. Por tanto, MoE requiere tanta o más memoria RAM/VRAM de hardware que un modelo denso de su tamaño total.
  • Inestabilidad en el entrenamiento: Si no se cuida, el Router puede aprender a enviar todos los tokens a un solo experto ("experto favorito"), dejando al resto sin entrenar. Se requieren pérdidas matemáticas de "balanceo" para forzar al Router a distribuir la carga uniformemente.

5. Arquitecturas híbridas (Mamba y SSM)

Aunque los Transformers dominan la IA actual, tienen un defecto de diseño fundamental: su costo de procesamiento crece de forma **cuadrática** con el largo del texto. Para solucionarlo, en 2023 se presentaron los **State Space Models (SSMs)** como Mamba. Hoy, los modelos más eficientes combinan ambas arquitecturas en diseños híbridos para obtener lo mejor de los dos mundos.

El problema del Transformer: Para calcular la atención de un texto de 100,000 palabras, el modelo debe hacer comparaciones cruzadas entre todas las palabras, lo que equivale a \(100,000 \times 100,000\) operaciónes. El procesamiento en tiempo lineal es el gran reto de la IA.

5.1 La intuición: El escritorio lleno vs. La taza de té

Compara cómo mantienen el contexto estas dos arquitecturas mientras leen un libro:

Transformer (Atención)

Mantiene **todas las páginas abiertas** en su escritorio. Puede mirar cualquier palabra del pasado con total precisión, pero a medida que el libro crece, se queda sin espacio físico en la mesa.

KV CacheCrece sin límite
ComplejidadO(N²) (Cuadrática)
Mamba / SSM (Estado de Espacio)

Mantiene **una sola taza de té**. Al leer cada página, toma un sorbo y el té absorbe la esencia de la historia. No guarda las páginas abiertas, solo actualiza el sabor del té constantemente.

Estado ocultoTamaño fijo
ComplejidadO(N) (Lineal)

5.2 ¿Cómo funciona un Modelo de Estado de Espacio (SSM)?

Inspirados en la teoría de sistemas de control en física e ingeniería, los SSMs procesan la información de manera matemática secuencial, similar a una Red Recurrente (RNN) pero con la capacidad de entrenarse en paralelo. Su ecuación matemática básica actualiza un **estado oculto (hidden state)** interno continuo:

# Ecuación básica de actualización de estado
h(t) = A * h(t-1) + B * x(t)
y(t) = C * h(t)
  • \(x(t)\): El token o palabra de entrada actual.
  • \(h(t)\): El estado interno (la "taza de té") que condensa la memoria del pasado.
  • \(y(t)\): La predicción o salida resultante.
  • \(A, B, C\): Matrices de transición que determinan cuánta memoria retener y qué ignorar. En **Mamba**, estas matrices cambian dinámicamente según la palabra de entrada (mecanismo de selección).

5.3 Arquitecturas Híbridas: Lo mejor de dos mundos

Modelos recientes de código abierto (como *Jamba* de AI21 o *RecurrentGemma* de Google) demuestran que no es necesario elegir uno. Se pueden intercalar capas de ambos en un solo modelo:

Capa Función Ventaja que aporta
Capas Mamba (SSM) Procesan la gran mayoría del texto en tiempo lineal \(O(N)\). Permite digerir contextos gigantescos a velocidades extremadamente altas.
Capas Transformer (Atención) Se activan ocasionalmente (ej: cada 4 u 8 capas). Aporta capacidad de recuperación exacta y razonamiento complejo (como recordar un número específico en la página 500).

5.4 Simulación conceptual: El bucle lineal vs. cuadrático

Este código en Python simula conceptualmente la diferencia en operaciónes necesarias para procesar un documento a medida que se hace más largo:

# Simulación del incremento de costo computacional
# N = número de tokens en el documento

def procesar_transformer(N):
    # Cuadrático: cada token se compara con todos los anteriores
    operaciones = 0
    for i in range(N):
        operaciones += i  # Comparación con tokens pasados
    return operaciones

def procesar_mamba(N, d_state=128):
    # Lineal: cada token solo actualiza el estado interno de tamaño fijo
    operaciones = 0
    for i in range(N):
        operaciones += d_state  # Multiplicación por matriz de estado fijo
    return operaciones

# Simulación con un libro de 5,000 palabras
tokens = 5000
ops_trans = procesar_transformer(tokens)
ops_mamba = procesar_mamba(tokens)

print(f"Costo de Atención (Transformer): {ops_trans:,} operaciones")
print(f"Costo de Estado (Mamba/SSM):    {ops_mamba:,} operaciones")
print(f"Mamba es ~{ops_trans / ops_mamba:.1f} veces más eficiente en computación")

5.5 Estado actual

El diseño híbrido se está convirtiendo en el estándar para modelos de contexto masivo de nueva generación. Al comprimir la información histórica en un estado oculto de tamaño constante, estos modelos reducen drásticamente el consumo de memoria RAM de video (VRAM) en inferencia, haciendo viable correr modelos potentes localmente en computadoras portátiles o teléfonos móviles.
