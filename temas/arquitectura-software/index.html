Arquitectura de Software

Piensa en la arquitectura de un edificio: antes de poner un solo ladrillo, hay planos, estructuras, columnas, instalaciones. La arquitectura de software es exactamente eso: el diseño estructural de un sistema antes de escribir una línea de código. Define cómo se organizan los componentes, cómo se comunican, y cómo evolucionan con el tiempo.

No es "cómo escribo esta función", sino "cómo organizo las 500 funciones para que no colapsen entre sí". Es la diferencia entre una choza de palos y un rascacielos: ambos refugian, pero uno aguanta un terremoto y el otro se cae con un estornudo.

La regla de oro: Una buena arquitectura retrasa las decisiones. En lugar de elegir tecnología X desde el día uno, diseñas capas que te permitan cambiar de opinión sin reescribir todo el sistema.

1. ¿Qué es Arquitectura de Software?

La arquitectura de software es el conjunto de decisiones de diseño que determinan la estructura de un sistema: sus componentes, sus relaciones, y las reglas que gobiernan su evolución. No es un documento que escribes una vez y olvidas; es una disciplina que guía cada línea de código.

Imagina que construyes una ciudad. No empiezas por las casas; primero diseñas calles, zonas residenciales, comerciales, industriales, suministro de agua y electricidad. La arquitectura de software hace lo mismo con el código: define los "barrios" (módulos), las "calles" (interfaces), y los "servicios públicos" (infraestructura).

Con buena arquitectura:

✓ Cambiar una parte no rompe otra
✓ Nuevos desarrolladores entienden rápido
✓ El sistema escala sin reescribirse
✓ Probar es sencillo y automatizable

Sin arquitectura:

✗ Cada cambio es una aventura
✗ Nadie sabe dónde va cada cosa
✗ Escalar es copiar y pegar todo
✗ Probar es manual y agotador

2. Estilos Arquitectónicos

Un estilo arquitectónico es una familia de sistemas con un patrón de organización común. Es como los estilos de construcción: gótico, barroco, modernista. Cada uno resuelve problemas distintos de forma distinta.

Monolítico

Todo el sistema en un solo ejecutable. Simple al inicio, complejo al crecer. Ideal para equipos pequeños y productos en fase temprana.

Microservicios

Cada funcionalidad es un servicio independiente. Escalan y se despliegan por separado. Requiere coordinación y experiencia en sistemas distribuidos.

Orientado a Eventos

Los componentes se comunican mediante eventos asíncronos. Ideal para sistemas que procesan flujos de datos en tiempo real.

Orientado a Capas

El sistema se divide en capas verticales (presentación, negocio, datos). Cada capa solo conoce a la inmediatamente inferior.

Orientado a Servicios (SOA)

Servicios reutilizables que se comunican a través de un bus empresarial (ESB). Precursor de los microservicios.

Arquitectura Hexagonal

El núcleo de la lógica de negocio está aislado del mundo exterior mediante puertos y adaptadores. También llamada Puertos y Adaptadores.

3. Patrones de Arquitectura

Los patrones arquitectónicos son soluciones probadas a problemas recurrentes de diseño de sistemas. A diferencia de los estilos (que son más amplios), los patrones son recetas específicas para problemas concretos.

Modelo-Vista-Controlador (MVC)

Divide la aplicación en tres partes: Modelo (datos y lógica), Vista (interfaz de usuario), y Controlador (mediador entre ambos). El controlador recibe entrada del usuario, actualiza el modelo, y la vista se actualiza automáticamente. Es el patrón más usado en aplicaciones web.

Repositorio (Repository)

Abstract la capa de persistencia para que la lógica de negocio no dependa de la base de datos concreta. Cambias de SQL a MongoDB sin tocar el código de negocio. Solo cambias la implementación del repositorio.

Inyección de Dependencias (DI)

En lugar de que una clase cree sus dependencias internamente, estas se las pasan desde fuera. Hace que el código sea más testeable y desacoplado. Es la base de frameworks como Spring, NestJS o Angular.

Observador (Observer)

Un objeto (sujeto) notifica a múltiples objetos (observadores) cuando cambia su estado. Es la base de los sistemas de eventos, reactividad y pub/sub.

Estrategia (Strategy)

Define una familia de algoritmos intercambiables. Por ejemplo, un sistema de notificaciones puede usar email, SMS o push según el contexto, sin modificar el código que las envía.

4. Arquitectura por Capas

Es el estilo más clásico. Divide la aplicación en capas horizontales donde cada una tiene una responsabilidad específica. La regla fundamental: una capa solo puede comunicarse con la capa inmediatamente inferior.

Ejemplo concreto: Un sistema de e-commerce donde la capa de presentación muestra productos (HTML/React), la capa de negocio calcula descuentos e impuestos, y la capa de persistencia guarda los pedidos en PostgreSQL. Si cambias de PostgreSQL a MySQL, solo tocas la capa de persistencia.

Ventajas

✓ Separación clara de responsabilidades
✓ Fácil de entender para equipos nuevos
✓ Cada capa puede probarse por separado
✓ Cambiar una capa no afecta a las demás

Desventajas

✗ Puede volverse rígida con el tiempo
✗ Cambios pequeños requieren tocar todas las capas
✗ El rendimiento sufre por tantas llamadas entre capas
✗ Tiende a crecer horizontalmente (capa de servicios enorme)

5. Monolítico vs Microservicios

El debate arquitectónico más famoso. No hay ganador absoluto: cada uno es mejor para contextos distintos.

Monolítico

Una sola aplicación que lo hace todo.

  • Un solo deploy para todo
  • Comunicación por llamadas a funciones (rápido)
  • Testing end-to-end sencillo
  • Escala vertical (más hardware)
  • Ideal para startups y equipos pequeños

Microservicios

Muchas aplicaciones pequeñas que colaboran.

  • Deploys independientes por servicio
  • Comunicación por red (HTTP, mensajería)
  • Cada servicio se prueba por separado
  • Escala horizontal (más instancias)
  • Ideal para equipos grandes y productos complejos

Consejo práctico: Empieza con un monolito bien estructurado. Divide en microservicios solo cuando el monolito demuestre ser un problema real. El 90% de los proyectos no necesitan microservicios desde el día uno.

6. Comunicación entre Componentes

Los componentes necesitan hablar entre sí. Hay dos grandes familias: síncrona y asíncrona. Elegir la correcta es clave para la escalabilidad.

Síncrona (HTTP/REST/gRPC)

  • El cliente espera la respuesta
  • Simple de implementar y depurar
  • Problema: si el servidor falla, el cliente falla
  • Problema: latencia acumulativa en cadenas de llamadas
  • Ideal para consultas y operaciones rápidas

Asíncrona (Colas, Eventos, Pub/Sub)

  • El cliente envía un mensaje y sigue
  • Más complejo de implementar
  • Ventaja: tolerancia a fallos natural
  • Ventaja: el productor no espera al consumidor
  • Ideal para procesos largos y desacoplamiento

Ejemplo: Cola de Mensajes con Redis


import redis
import json
import time

# Conexión a Redis
cola = redis.Redis(host='localhost', port=6379, db=0)

# --- Productor (Servicio de Pedidos) ---
def crear_pedido(usuario_id, producto, cantidad):
    pedido = {
        "usuario": usuario_id,
        "producto": producto,
        "cantidad": cantidad,
        "timestamp": time.time()
    }
    # Enviar pedido a la cola "pedidos_pendientes"
    cola.lpush("pedidos_pendientes", json.dumps(pedido))
    print(f"Pedido encolado: {producto} x {cantidad}")
    return {"status": "en_cola"}

# --- Consumidor (Servicio de Inventario) ---
def procesar_pedidos():
    while True:
        # Esperar por un pedido (bloqueante hasta que llegue)
        _, dato = cola.brpop("pedidos_pendientes")
        pedido = json.loads(dato)
        print(f"Procesando pedido: {pedido['producto']}")
        # Simular descuento de inventario
        time.sleep(2)
        print(f"Inventario actualizado para {pedido['usuario']}")

# El productor y el consumidor corren en procesos separados

7. Principios SOLID

SOLID son 5 principios de diseño orientado a objetos que te ayudan a escribir código mantenible y escalable. No son reglas, sino guías. Violarlos tiene consecuencias, pero a veces vale la pena.

S — Single Responsibility (Responsabilidad Única)

Una clase debe tener una sola razón para cambiar. Si tu clase Pedido calcula impuestos, envía emails y guarda en BD, tienes tres razones para cambiarla. Sepáralo en tres clases.

O — Open/Closed (Abierto/Cerrado)

Las clases deben estar abiertas para extensión, cerradas para modificación. Agrega funcionalidad nueva con herencia o composición, no modificando el código existente.

L — Liskov Substitution (Sustitución de Liskov)

Si una clase B extiende a A, deberías poder usar B donde esperas A sin que el programa falle. Una subclase no debe romper el comportamiento de la superclase. Por ejemplo, una clase Rectángulo no debería heredar de Cuadrado solo porque "matemáticamente tiene sentido".

I — Interface Segregation (Segregación de Interfaces)

Es mejor tener muchas interfaces pequeñas y específicas que una interfaz grande y general. Si una clase implementa una interfaz con métodos que no usa, la interfaz es demasiado grande.

D — Dependency Inversion (Inversión de Dependencias)

Los módulos de alto nivel no deben depender de módulos de bajo nivel. Ambos deben depender de abstracciones. En criollo: programa contra interfaces, no contra implementaciones concretas.

8. Arquitectura Hexagonal (Puertos y Adaptadores)

También conocida como Puertos y Adaptadores, fue propuesta por Alistair Cockburn. La idea central es aislar la lógica de negocio del mundo exterior. Piensa en un timón de barco: el timón (lógica de negocio) no cambia aunque cambies el barco (base de datos, interfaz de usuario, API).

La arquitectura hexagonal divide el sistema en tres zonas:

Núcleo (Dominio)

La lógica de negocio pura. No sabe nada de HTTP, bases de datos, ni frameworks. Solo contiene entidades, reglas de negocio y puertos (interfaces).

Puertos (Interfaces)

Contratos que definen cómo el núcleo se comunica con el exterior. Por ejemplo: RepositorioUsuario, ServicioNotificaciones.

Adaptadores

Implementaciones concretas de los puertos. Un adaptador PostgreSQL, otro para MongoDB, otro para una API REST, otro para una interfaz de línea de comandos.

Ejemplo real: Tu sistema de pagos tiene un puerto ProcesadorPagos. Implementas un adaptador para Stripe. Seis meses después quieres agregar PayPal. Creas otro adaptador que implementa el mismo puerto. El núcleo no cambia ni una línea.

9. CQRS y Event Sourcing

CQRS (Command Query Responsibility Segregation) separa las operaciones de lectura de las de escritura. En lugar de un solo modelo para leer y escribir, tienes dos: uno optimizado para consultas (reads) y otro para comandos (writes).

Event Sourcing guarda el estado como una secuencia de eventos en lugar de un estado actual. En lugar de "el saldo es 100", guardas "se depositó 50, se retiró 20, se depositó 70". El saldo es la suma de todos los eventos.

CRUD tradicional

Un solo modelo para leer y escribir.

  • UPDATE reemplaza el estado anterior
  • No hay historial de cambios
  • Simple, pero inflexible

CQRS + Event Sourcing

Modelos separados, historial completo.

  • Cada cambio es un evento inmutable
  • Historial completo de auditoría
  • Puedes reconstruir el estado en cualquier momento

# Event Sourcing: guardamos eventos, no estado
eventos = []

def depositar(cuenta_id, monto):
    eventos.append({
        "tipo": "DEPOSITO",
        "cuenta": cuenta_id,
        "monto": monto
    })

def retirar(cuenta_id, monto):
    eventos.append({
        "tipo": "RETIRO",
        "cuenta": cuenta_id,
        "monto": monto
    })

def calcular_saldo(cuenta_id):
    saldo = 0
    for e in eventos:
        if e["cuenta"] == cuenta_id:
            if e["tipo"] == "DEPOSITO":
                saldo += e["monto"]
            elif e["tipo"] == "RETIRO":
                saldo -= e["monto"]
    return saldo

# Uso
depositar("A1", 100)
retirar("A1", 30)
depositar("A1", 50)
print(calcular_saldo("A1"))  # 120
# Podemos ver el historial completo de la cuenta

10. Atributos de Calidad (No Funcionales)

La arquitectura no solo se preocupa de que el software funcione, sino de cómo funciona. Estos son los atributos de calidad que toda arquitectura debe considerar:

Rendimiento

Qué tan rápido responde el sistema. Se mide en latencia (tiempo por petición) y throughput (peticiones por segundo).

Escalabilidad

Capacidad de manejar más carga agregando recursos. Escalar vertical (más CPU/RAM) vs horizontal (más servidores).

Disponibilidad

Porcentaje de tiempo que el sistema está operativo. Un sistema "cinco nueves" (99.999%) tolera ~5 minutos de caída al año.

Seguridad

Protección contra accesos no autorizados, fugas de datos y ataques. Autenticación, autorización, cifrado, auditoría.

Mantenibilidad

Qué tan fácil es modificar el sistema. Código limpio, pruebas automatizadas, documentación clara, baja deuda técnica.

Tolerancia a Fallos

Capacidad de seguir funcionando cuando algo falla. Redundancia, circuit breakers, reintentos, degradación gradual.
