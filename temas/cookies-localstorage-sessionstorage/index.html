Cookies, Local Storage y Session Storage

En resumen: Tu navegador es un cajón con tres compartimentos diferentes para guardar datos. Cada uno tiene reglas distintas sobre quién puede abrirlo, cuándo se vacía y qué tan peligroso es si alguien más accede a él.

1. El problema que resuelven

HTTP es un protocolo sin memoria. Cada petición que haces a un servidor es como si fuera la primera vez que te ven. No recuerdan quién eres, qué hiciste antes ni qué preferencias tenías.

Las cookies, local storage y session storage existen para darle memoria al navegador. Son mecanismos de almacenamiento del lado del cliente que permiten persistir datos entre peticiones.

Cookies

Pequeños textos que el servidor pide al navegador que guarde. Se envían automáticamente con cada petición HTTP al mismo dominio.

Local Storage

Almacén key-value que persiste indefinidamente. Solo accesible por JavaScript en el navegador. Nunca se envía al servidor automáticamente.

Session Storage

Igual que local storage pero se borra automáticamente cuando cierras la pestaña. Un almacén efímero por pestaña.

2. Cookies: el mecanismo antiguo y problemático

Las cookies nacieron en 1994 en Netscape. La idea era simple: el servidor envía un encabezado Set-Cookie con la respuesta HTTP, y el navegador guarda ese dato. A partir de ahí, el navegador adjunta automáticamente ese cookie en cada petición subsiguiente al mismo dominio.

Imagínalo como un ticket de entrada a un evento. El organizador te da un ticket (cookie) y cada vez que pasas por una puerta (petición HTTP), lo muestras. No tienes que decir quién eres cada vez: el ticket lo hace por ti.


# El servidor envía una cookie en la respuesta HTTP
HTTP/1.1 200 OK
Set-Cookie: usuario_id=12345; Path=/; HttpOnly; Secure; SameSite=Strict

# El navegador la envía automáticamente en siguientes peticiones
GET /api/perfil HTTP/1.1
Cookie: usuario_id=12345

2.1 Tipos de cookies

Cookies de sesión: No tienen fecha de expiración. El navegador las borra cuando cierras toda la ventana (no solo la pestaña).

Cookies persistentes: Tienen un Max-Age o Expires. Sobreviven al cierre del navegador.


// Establecer una cookie desde JavaScript
document.cookie = "tema=oscuro; max-age=31536000; path=/";

// Esto NO sobreescribe las demás cookies, solo agrega o modifica una
// Para borrar una cookie, ponle max-age en 0:
document.cookie = "tema=oscuro; max-age=0; path=/";

2.2 Atributos de seguridad

Estos atributos son la única defensa que tienes contra el mal uso de las cookies:

  • HttpOnly — JavaScript no puede leer esta cookie (document.cookie la ignora). Protege contra ataques XSS.
  • Secure — Solo se envía por conexiones HTTPS. Nunca por HTTP.
  • SameSite=Strict|Lax|None — Controla si la cookie se envía en peticiones cruzadas (cross-site). Es la defensa principal contra CSRF.
  • Path — Define a qué rutas del sitio aplica la cookie.
  • Domain — Define en qué dominios se envía la cookie.

3. Local Storage: el almacén permanente

Local storage es parte de la API Web Storage que apareció con HTML5. Es un par de clave-valor que vive en el navegador, es accesible solo por JavaScript del mismo origen, y nunca se envía al servidor.

La diferencia clave con las cookies: local storage es un almacén privado entre tú y tu navegador. El servidor no lo ve a menos que explicitamente lo leas y lo envíes en una petición.


// Guardar datos
localStorage.setItem("preferencias", JSON.stringify({
  tema: "oscuro",
  idioma: "es",
  notificaciones: false
}));

// Leer datos
const prefs = JSON.parse(localStorage.getItem("preferencias"));

// Eliminar un valor
localStorage.removeItem("preferencias");

// Vaciar todo
localStorage.clear();

// Acceder por índice (menos común)
const clave = localStorage.key(0);

Capacidad: ~5-10 MB por dominio (varía entre navegadores). Mucho más que las cookies (4 KB por cookie).

3.1 Lifecycle

setItem() → Persiste en disco → Disponible siempre → Hasta que lo borres

Local storage no expira. Si el usuario limpia los datos del navegador o se borran explícitamente, se pierde. No hay mecanismo de expiración automático.

4. Session Storage: el almacén efímero

Session storage funciona casi igual que local storage, pero con una restricción: se borra cuando cierras la pestaña. Cada pestaña tiene su propio session storage, aislado de las demás.


// Funciona igual que localStorage
sessionStorage.setItem("carrito", JSON.stringify([
  { id: 1, cantidad: 2 },
  { id: 3, cantidad: 1 }
]));

// Pero se pierde al cerrar la pestaña
const carrito = JSON.parse(sessionStorage.getItem("carrito"));

Es útil para datos temporales de una sesión de usuario que no necesitan persistir: formularios multi-paso, estado transitorio de una compra, datos de una encuesta.

6. Por qué las cookies son peligrosas

Las cookies no son inherentemente malas, pero tienen un diseño que las hace vulnerables a varios ataques. La razón principal es que se envían automáticamente sin que el usuario lo sepa ni lo autorice explícitamente.

  • Cross-Site Scripting (XSS): Si un atacante inyecta JavaScript malicioso en tu sitio (por ejemplo, a través de un comentario no sanitizado), puede leer todas las cookies que no tengan el atributo HttpOnly. Con ellas puede secuestrar la sesión del usuario.
  • Cross-Site Request Forgery (CSRF): Un sitio malicioso puede hacer peticiones a tu bank.com mientras el usuario tiene una sesión activa. El navegador adjunta la cookie automáticamente porque la petición va al mismo dominio. El banco no puede distinguir entre una petición legítima y una forzada.
  • Tracking / Vigilancia: Los cookies de terceros (third-party cookies) permiten que un dominio como tracker.com te siga por toda la web. Cada página que visitas que tiene un script de tracker.com envía la cookie, creando un perfil de tu comportamiento.
  • Robo de sesión: Si interceptan la cookie (por ejemplo, en una red WiFi pública sin HTTPS), pueden suplantar al usuario completamente. El cookie es el ticket de entrada: quien lo tiene, entra.
  • Datos sin cifrar: Las cookies se almacenan en texto plano en el disco del usuario. Cualquier programa con acceso al sistema de archivos puede leerlas.

6.1 Local storage es "más seguro" pero no es infalible

Local storage no se envía automáticamente al servidor, lo que elimina los ataques CSRF. Pero sigue siendo vulnerable a XSS: si un atacante inyecta JavaScript en tu página, puede leer todo lo que guardaste en local storage.

La diferencia es sutil pero importante: con cookies el servidor puede aplicar atributos de seguridad (HttpOnly, SameSite). Con local storage, la única defensa es prevenir que el JavaScript malicioso se ejecute en primer lugar.

7. ¿Cuándo usar cada uno?

Regla simple: Si el servidor necesita el dato, usa cookies. Si solo lo necesita el navegador, usa local storage o session storage.

Cookies: Autenticación de sesión, preferencias que el servidor debe conocer (idioma, moneda), tokens de seguridad (CSRF tokens).

Local Storage: Preferencias de UI (tema oscuro, sidebar colapsada), caché de datos offline, estado de una aplicación SPA que no sale del navegador.

Session Storage: Datos temporales de formularios, estado intermedio de un proceso multi-paso, datos que no deben compartirse entre pestañas.

8. Alternativas modernas

Los navegadores están migrando hacia modelos más seguros:

  • HttpOnly + Secure + SameSite cookies — La defensa estándar para autenticación. La mayoría de frameworks web las configuran así por defecto.
  • API Storage — No confundir con Web Storage. Es una API más moderna que soporta estructuras de datos más complejas y operaciones asíncronas.
  • IndexedDB — Base de datos completa en el navegador. Mucho más potente que localStorage para cantidades grandes de datos.
  • Cache API + Service Workers — Para cachear respuestas HTTP y funcionar offline. Es el modelo que usan las Progressive Web Apps (PWAs).

// IndexedDB: base de datos en el navegador
const request = indexedDB.open("MiBaseDatos", 1);

request.onupgradeneeded = (e) => {
  const db = e.target.result;
  const store = db.createObjectStore("productos", { keyPath: "id" });
  store.createIndex("nombre", "nombre", { unique: false });
};

request.onsuccess = (e) => {
  const db = e.target.result;
  const tx = db.transaction("productos", "readwrite");
  tx.objectStore("productos").add({ id: 1, nombre: "Laptop", precio: 999 });
};
