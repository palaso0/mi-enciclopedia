Testing: Unit, Integration, E2E y Regresión

En resumen: Los tests son código que verifica que tu código funciona. Unit tests prueban piezas aisladas, integration tests prueban cómo se juntan, E2E tests simulan un usuario real. Todos son necesarios, pero en diferentes proporciones.

1. La pirámide de testing

La pirámide de testing te dice cuántos tests de cada tipo deberías tener. La base (más tests) son los unit, la punta (menos tests) son los E2E.

E2E
Pocos: lentos, frágiles, costosos
Integration
Algunos: prueban la integración real
Unit
Muchos: rápidos, baratos, confiables

2. Unit Tests

Unit Tests

Prueban una función o módulo de forma aislada. No dependen de base de datos, red ni otros servicios. Si tu función tiene dependencias, se mockean.

rápidos baratos confiables 

// Función a testear
function sumar(a, b) {
  return a + b;
}

// Unit test (con Jest/Vitest)
test("sumar dos números positivos", () => {
  expect(sumar(2, 3)).toBe(5);
});

test("sumar con negativos", () => {
  expect(sumar(-1, -3)).toBe(-4);
});

test("sumar con cero", () => {
  expect(sumar(5, 0)).toBe(5);
});

Cuándo usar mocks


// Función que depende de una API externa
async function obtenerUsuario(id) {
  const res = await fetch(`/api/usuarios/${id}`);
  return res.json();
}

// Test con mock: no hacemos petición real
test("obtenerUsuario retorna los datos", async () => {
  global.fetch = jest.fn().mockResolvedValue({
    json: () => Promise.resolve({ id: 1, nombre: "Ana" })
  });

  const usuario = await obtenerUsuario(1);
  expect(usuario.nombre).toBe("Ana");
  expect(fetch).toHaveBeenCalledWith("/api/usuarios/1");
});

3. Integration Tests

Integration Tests

Prueban cómo funcionan varias piezas juntas. Por ejemplo: un endpoint de API que consulta la base de datos, procesa los datos y retorna una respuesta.

más realistas más lentos más costosos 

// Integration test de un endpoint (con Supertest)
const request = require("supertest");
const app = require("../app");
const db = require("../db");

beforeAll(async () => {
  await db.migrate.latest();
  await db.seed.run();
});

afterAll(async () => {
  await db.destroy();
});

test("GET /api/usuarios retorna la lista", async () => {
  const res = await request(app).get("/api/usuarios");
  expect(res.status).toBe(200);
  expect(res.body.length).toBeGreaterThan(0);
});

test("POST /api/usuarios crea un usuario", async () => {
  const res = await request(app)
    .post("/api/usuarios")
    .send({ nombre: "Carlos", email: "carlos@test.com" });
  expect(res.status).toBe(201);
  expect(res.body.nombre).toBe("Carlos");
});

4. End-to-End (E2E) Tests

End-to-End Tests

Simulan un usuario real usando la aplicación. Abran un navegador, hacen clic, escriben, navegan. Prueban el sistema completo de punta a punta.

máxima confianza muy lentos muy costosos frágiles 

// E2E test con Playwright
const { test, expect } = require("@playwright/test");

test("login funciona correctamente", async ({ page }) => {
  await page.goto("https://tu-sitio.com/login");

  await page.fill('[data-testid="email"]', "usuario@test.com");
  await page.fill('[data-testid="password"]', "123456");
  await page.click('[data-testid="boton-login"]');

  await expect(page).toHaveURL("/dashboard");
  await expect(page.locator("h1")).toContainText("Bienvenido");
});

test("carrito de compras funciona", async ({ page }) => {
  await page.goto("https://tu-sitio.com/productos");
  await page.click('[data-testid="agregar-producto-1"]');
  await page.click('[data-testid="ver-carrito"]');

  await expect(page.locator(".carrito-item")).toHaveCount(1);
  await expect(page.locator(".total")).toContainText("$999");
});

5. Tests de Regresión

Tests de Regresión

No son un tipo de test separado, sino el propósito de todos los tests. Son tests que verifican que un bug que se corrigió no vuelva a aparecer.

Si encontrás un bug, escribís un test que lo reproduzca. Cuando lo arreglás, el test pasa. Si el bug vuelve en el futuro, el test falla.


// Bug: la función no manejaba strings vacíos
function formatearNombre(nombre) {
  if (!nombre) return "Sin nombre";
  return nombre.charAt(0).toUpperCase() + nombre.slice(1);
}

// Test de regresión: este test evita que el bug vuelva
test("regresión: formatearNombre con string vacío", () => {
  expect(formatearNombre("")).toBe("Sin nombre");
  expect(formatearNombre(null)).toBe("Sin nombre");
  expect(formatearNombre(undefined)).toBe("Sin nombre");
});

6. Métricas de testing

  • Cobertura de código: Qué porcentaje de tu código está cubierto por tests. No es la única métrica, pero es un buen indicador.
  • Cobertura de ramas: Qué porcentaje de las ramas del código (if/else, switch) se ejecutaron en los tests.
  • Mutación testing: Cambiá tu código a propósito (mutaciones) y verificá que los tests fallen. Si un test no detecta la mutación, no está sirviendo.

// Cobertura en Jest/Vitest
// npx jest --coverage
//
// Statements   : 85% ( 170/200 )
// Branches     : 78% ( 50/64 )
// Functions    : 90% ( 18/20 )
// Lines        : 85% ( 165/194 )

7. Anti-patrones en testing

  • Tests que dependen de otros tests: Cada test debe poder ejecutarse independientemente.
  • Tests frágiles: Si un test falla porque cambiaste el color de un botón, no es un buen test.
  • Tests lentos en development: Los tests unitarios deben correr en milisegundos. Si tardan mucho, los desarrolladores no los corren.
  • Testear implementación en vez de comportamiento: No testees que se llamó cierta función, testees que el resultado sea el esperado.
  • Cobertura del 100% como meta: No todo el código vale la pena testearlo. Enfocá los tests en la lógica crítica.

La regla práctica: Empezá con unit tests para la lógica de negocio. Agregá integration tests para los endpoints y servicios críticos. Usá E2E tests para los flujos principales del usuario. No intentés testear todo: enfocá los esfuerzos donde generan más valor.
