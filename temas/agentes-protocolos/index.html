Agentes y Ecosistemas

Los agentes de IA representan el salto más importante en la evolución de los modelos de lenguaje: de simples generadores de texto a sistemas autónomos que observan, razonan y actúan sobre el mundo real. Este mega-tema recorre los cuatro pilares que conforman el ecosistema moderno de agentes: el bucle de decisión-acción que los gobierna, el function calling que les permite usar herramientas, las skills que encapsulan conocimiento reutilizable, y el Model Context Protocol que estandariza cómo se conectan con el mundo exterior.

5.1 Agentes de IA: El loop de decisión-acción

Un chatbot te responde. Un agente hace cosas por ti. La diferencia es fundamental: mientras un chatbot genera texto y se detiene, un agente observa su entorno, toma decisiones y ejecuta acciónes reales hasta completar un objetivo. Piensa en un asistente personal que no sólo conversa, sino que busca información, ejecuta comandos, gestiona tu calendario y coordina con otros servicios para lograr lo que le pediste.

La analogía clave: Un chatbot es como un diccionario: puedes preguntarle cosas y te da respuestas. Un agente de IA es como un asistente personal competente: le das un objetivo ("organiza mi viaje a Barcelona"), y él coordina reservas, busca vuelos, verifica disponibilidad y te entrega un plan completo. La diferencia no está en la inteligencia, sino en la capacidad de actuar.

1. La diferencia entre un chatbot y un agente

Un chatbot tradicional opera en un ciclo simple: recibe una pregunta y genera una respuesta. No tiene acceso a herramientas externas, no mantiene memoria entre interacciónes, y no puede ejecutar acciónes en el mundo real. Un agente de IA rompe esas limitaciones al integrar tres capacidades adicionales:

Capacidad Chatbot Agente
Generación de texto Sí Sí
Acceso a herramientas No Sí (APIs, bases de datos, código)
Memoria persistente No (sólo ventana actual) Sí (corto y largo plazo)
Planificación multi-paso No Sí (descompone tareas complejas)
Ejecución autónoma No (espera input humano) Sí (bucle decisión-acción)

2. El bucle Observe-Piensa-Actúa

Todo agente de IA gira alrededor de un ciclo fundamental: observar, pensar y actuar. Este bucle se repite iterativamente hasta que el agente considera que la tarea está completa.

Paso 1
OBSERVAR — Recopilar información del entorno
↓
Paso 2
PENSAR — Analizar, planificar y decidir la siguiente acción
↓
Paso 3
ACTUAR — Ejecutar la acción mediante una herramienta
↻
¿Objetivo completado? Si no → repetir

Cada paso del bucle tiene un rol claro: Observar es recopilar datos (leer un archivo, consultar una API, recibir la respuesta del usuario). Pensar es razonar sobre qué hacer con esos datos (el LLM decide qué herramienta usar y con qué parámetros). Actuar es ejecutar la herramienta seleccionada y obtener un resultado.

3. Memoria: Corto plazo y largo plazo

Un agente sin memoria es como un employado que olvida todo lo que le dijiste cada vez que te vas de la habitación. Los agentes modernos manejan dos tipos de memoria que les permiten mantener contexto y aprender:

Memoria a corto plazo

Es el contexto de la conversación actual. Se almacena en la ventana de contexto del LLM (el "buffer" de tokens). Cuando se llena, los datos más antiguos se descartan. Es rápida pero limitada en capacidad.

Memoria a largo plazo

Es información persistente guardada externamente: bases de vectores, archivos, bases de datos. Permite al agente recordar interacciónes pasadas, preferencias del usuario y conocimiento acumulado entre sesiones.

La memoria a largo plazo funciona de manera similar a un RAG (Retrieval-Augmented Generation): el agente almacena información en un vector store y la recupera por similitud cuando la necesita. Esto le permite mantener contexto sin saturar la ventana de tokens.

4. Estrategias de planificación

No todas las tareas se pueden resolver con un sólo paso. Los agentes necesitan estrategias para descomponer problemas complejos en pasos manejables:

  • Planificación reactiva: El agente toma una acción a la vez, evalúa el resultado y decide el siguiente paso. Es simple pero puede ser lento para tareas complejas.
  • Planificación jerárquica: El agente genera un plan completo antes de ejecutar, con sub-tareas y dependencias. Más eficiente pero requiere mayor capacidad de razonamiento.
  • Planificación por reflejo: El agente tiene reglas predefinidas que activan ciertas acciónes bajo ciertas condiciones. Rápido pero inflexible.
  • Planificación con herramientas: El agente delega partes de la planificación a herramientas especializadas (por ejemplo, un planificador que genera un DAG de tareas).

5. Sistemas multi-agente y frameworks reales

Los agentes más poderosos no trabajan solos. Los sistemas multi-agente coordinan varios agentes especializados, cada uno con un rol específico, para resolver problemas que un sólo agente no podría abordar eficientemente.

LangChain / LangGraph

Framework modular para construir agentes con grafo de estados. Permite definir nodos (acciónes) y aristas (transiciones) para crear flujos complejos con control total.

AutoGPT

Uno de los primeros agentes autónomos de código abierto. Genera sus propios prompts y ejecuta tareas de forma iterativa sin intervención humana.

CrewAI

Framework para orquestar equipos de agentes con roles definidos (investigador, escritor, revisor). Cada agente tiene herramientas y objetivos específicos.

OpenAI Assistants

API gestionada que proporciona agentes con herramientas integradas (code interpreter, file search, function calling). Abstracta la complejidad del loop.

6. Simulación: Un agente en acción

Veamos cómo se ve el bucle de un agente en pseudocódigo simplificado. Imagina un agente que debe investigar un tema y crear un resumen:

class AgenteIA:
    def __init__(self, llm, herramientas, memoria):
        self.llm = llm                # El modelo de lenguaje (cerebro)
        self.herramientas = herramientas  # Dict de herramientas disponibles
        self.memoria = memoria        # Almacén de corto y largo plazo
        self.historial = []           # Registro de acciones tomadas

    def ejecutar(self, objetivo):
        """Bucle principal: Observe → Piensa → Actúa"""
        while not self.objetivo_cumplido(objetivo):
            # 1. OBSERVAR: Recopilar contexto actual
            contexto = self.construir_contexto(objetivo)
            
            # 2. PENSAR: El LLM decide la siguiente acción
            decision = self.llm.predict(f"""
                Objetivo: {objetivo}
                Contexto: {contexto}
                Historial: {self.historial}
                ¿Qué herramienta debo usar y con qué parámetros?
            """)
            
            # 3. ACTUAR: Ejecutar la herramienta elegida
            herramienta = self.herramientas[decision.nombre]
            resultado = herramienta.ejecutar(decision.parametros)
            
            # Guardar en memoria
            self.historial.append({
                "accion": decision.nombre,
                "parametros": decision.parametros,
                "resultado": resultado
            })
            
            # Evaluar si el objetivo se cumplió
            if self.es_suficiente(resultado, objetivo):
                return self.sintetizar_respuesta()
        
        return "Se alcanzó el límite de iteraciones"

Nota sobre la implementación real: En la práctica, los frameworks como LangChain ya implementan este bucle internamente. El desarrollador define las herramientas, el objetivo y la estrategia de planificación, y el framework se encarga del ciclo observe-piensa-actúa. Lo importante es entender el patrón subyacente para poder diseñar, depurar y mejorar agentes efectivos.

5.2 Function Calling: Uso de herramientas

Un LLM por sí sólo sólo puede generar texto. Function calling es el mecanismo que le permite decir "necesito usar esta herramienta con estos datos" en lugar de simplemente inventar una respuesta. Es la diferencia entre un modelo que te dice "hace calor hoy" y uno que consulta una API del clima y te da la temperatura exacta de tu ciudad.

La analogía del restaurante: Piensa en un LLM como un cliente en un restaurante. El cliente sabe qué quiere comer (el objetivo), pero no puede cocinar. Le dice al mesero (el sistema): "Quiero una pasta al pesto sin frutos secos". El mesero traduce eso a una orden específica (function call) para la cocina (API externa), la cocina prepara el plato, y el mesero regresa con el resultado (la respuesta de la función). El cliente nunca entra a la cocina — sólo recibe el resultado.

1. ¿Cómo funciona el mecanismo?

El proceso de function calling sigue un flujo preciso entre el usuario, el LLM y las herramientas externas:

👤
Usuario "¿Qué tiempo hace en Buenos Aires mañana?"
↓
🧠
LLM analiza Identifica que necesita datos externos. Decide llamar a obtener_clima(ciudad="Buenos Aires", fecha="mañana")
↓
⚡
Sistema ejecuta Envía la llamada a la API del clima con los parámetros exactos
↓
📡
API responde Retorna: {"temp": 22, "condicion": "parcialmente nublado", "lluvia": 30}
↓
🧠
LLM integra "Mañana en Buenos Aires habrá 22°C con cielo parcialmente nublado. Hay 30% de probabilidad de lluvia, llevá paraguas."

Punto clave: El LLM nunca ejecuta la función directamente. Sólo genera la llamada con el nombre correcto y los parámetros correctos. Es el sistema externo (el runtime del agente) quien realmente conecta con la API y ejecuta la operación.

2. Formato de la llamada: Request y Response

Cada function call tiene un formato estructurado. Primero, el desarrollador define las herramientas disponibles para el modelo. Luego, cuando el modelo decide usar una herramienta, genera un JSON con la estructura esperada:

Paso 1: Definición de herramientas 
{
  "nombre": "obtener_clima",
  "descripcion": "Obtiene el pronóstico del clima para una ciudad",
  "parametros": {
    "type": "object",
    "properties": {
      "ciudad": {
        "type": "string",
        "description": "Nombre de la ciudad"
      },
      "fecha": {
        "type": "string",
        "enum": ["hoy", "mañana", "pasado"],
        "description": "Para qué día se consulta"
      }
    },
    "required": ["ciudad"]
  }
}
Paso 2: Respuesta del modelo (function call) 
{
  "tool_calls": [
    {
      "id": "call_abc123",
      "function": {
        "name": "obtener_clima",
        "arguments": "{\"ciudad\": \"Buenos Aires\", \"fecha\": \"mañana\"}"
      }
    }
  ]
}
Paso 3: Resultado devuelto al modelo 
{
  "tool_call_id": "call_abc123",
  "output": {
    "ciudad": "Buenos Aires",
    "temperatura": 22,
    "condicion": "parcialmente nublado",
    "probabilidad_lluvia": 30,
    "viento_kmh": 15
  }
}

3. Validación de parámetros

El sistema valida que los parámetros generados por el modelo coincidan con el esquema definido. Esto previene errores en tiempo de ejecución:

Parámetro Tipo Requerido Validación
ciudad string Sí No puede estar vacío
fecha enum No Debe ser "hoy", "mañana" o "pasado"
unidades enum No Debe ser "celsius" o "fahrenheit"

Si el modelo genera un parámetro inválido (por ejemplo, una fecha fuera del enum), el sistema puede rechazar la llamada antes de ejecutarla, o devolver un error que el modelo puede usar para corregir su respuesta.

4. Llamadas encadenadas y ejecución paralela

Las tareas complejas suelen requerir múltiples llamadas a herramientas. Hay dos patrones principales:

Encadenamiento secuencial: Cada llamada depende del resultado de la anterior. El modelo genera una llamada, recibe el resultado, decide la siguiente, y así sucesivamente.

buscar_usuario("pablo")
→
obtener_pedidos(id=42)
→
calcular_total(pedidos)

Ejecución paralela: Cuando las llamadas son independientes, el modelo puede generar múltiples tool calls en una sola respuesta. El sistema las ejecuta simultáneamente y devuelve todos los resultados juntos.

obtener_clima("Buenos Aires")
+
obtener_clima("Córdoba")
+
obtener_clima("Mendoza")

5. Ejemplo real: Consulta a una base de datos

Veamos un ejemplo completo de cómo un asistente de e-commerce usa function calling para responder preguntas sobre pedidos:

import json

# Herramientas disponibles para el asistente
herramientas = [
    {
        "nombre": "buscar_producto",
        "descripcion": "Busca productos por nombre o categoría",
        "parametros": {
            "consulta": "string (requerido)",
            "categoria": "string (opcional)",
            "precio_maximo": "number (opcional)"
        }
    },
    {
        "nombre": "consultar_stock",
        "descripcion": "Verifica disponibilidad de un producto",
        "parametros": {
            "producto_id": "string (requerido)",
            "sucursal": "string (opcional)"
        }
    },
    {
        "nombre": "crear_pedido",
        "descripcion": "Crea un nuevo pedido con los productos indicados",
        "parametros": {
            "productos": "array de {id, cantidad} (requerido)",
            "direccion_envio": "string (requerido)"
        }
    }
]

# Simulación: el usuario hace una pregunta
pregunta = "¿Tienen auriculares bluetooth por menos de $50.000?"

# El LLM decide qué herramienta usar
decision_llm = {
    "tool_calls": [{
        "function": {
            "name": "buscar_producto",
            "arguments": json.dumps({
                "consulta": "auriculares bluetooth",
                "precio_maximo": 50000
            })
        }
    }]
}

# El sistema ejecuta la función
resultado_real = buscar_producto(
    consulta="auriculares bluetooth",
    precio_maximo=50000
)
# → [{"id": "AUR001", "nombre": "Sony WH-CH520", "precio": 35000},
#    {"id": "AUR002", "nombre": "JBL Tune 510BT", "precio": 28000}]

# El LLM integra el resultado en una respuesta natural
respuesta = "Sí, tenemos dos opciones por debajo de $50.000:
  1. Sony WH-CH520 — $35.000
  2. JBL Tune 510BT — $28.000
  ¿Querés que verifique el stock de alguno?"

La ventaja de function calling: El LLM no "inventa" precios ni inventarios. Consulta datos reales del sistema y genera una respuesta precisa y actualizada. Si no encuentra resultados, simplemente lo informa en lugar de fabricar información.

6. Consideraciones importantes

Function calling no es infalible. Hay varios aspectos a tener en cuenta al diseñar un sistema con herramientas:

  • El modelo puede equivocarse: A veces genera parámetros incorrectos o elige la herramienta equivocada. Siempre validar las llamadas antes de ejecutarlas.
  • Seguridad: Si una función puede modificar datos o ejecutar código, el agente necesita permisos bien definidos. Nunca expongas funciones destructivas sin validación humana.
  • Latencia: Cada llamada a herramienta agrega tiempo. Las llamadas paralelas mitigan esto, pero el encadenamiento secuencial puede ser lento.
  • Límites de la ventana: El resultado de cada función consume tokens en la ventana de contexto. Si los resultados son muy grandes, el modelo puede perder información importante.

5.3 Skills: Contexto reutilizable para agentes

Un chef experimentado no memoriza cada receta desde cero cada vez que cocina. Tiene un libro de recetas: cada receta es un conjunto de instrucciónes que puede consultar, adaptar y ejecutar en cualquier momento. Las skills en agentes de IA funcionan igual: son paquetes reutilizables de instrucciónes, contexto y herramientas que un agente puede cargar dinámicamente según la tarea que necesite resolver.

La analogía del chef: Un chef sin libro de recetas tendría que reinventar cada platillo desde cero cada vez. Un chef con un libro puede preparar cualquier receta que conozca, combinar sabores de diferentes recetas, e incluso crear nuevas variantes a partir de las que ya tiene. Las skills son ese libro de recetas para agentes de IA: cada skill encapsula un conjunto de conocimientos e instrucciónes que el agente puede activar bajo demanda.

1. ¿Qué es una skill exactamente?

Una skill es un módulo reutilizable que contiene tres componentes esenciales:

Instrucciones

Prompt o conjunto de directrices que definen cómo el agente debe abordar un tipo específico de tarea. Incluye formato de salida, restricciones y mejores prácticas.

Herramientas

Funciones o APIs específicas que el agente puede invocar cuando esta skill está activa. No todas las skills necesitan herramientas, pero la mayoría sí.

Contexto

Ejemplos, plantillas, datos de referencia y conocimiento especializado que el agente necesita para ejecutar la skill correctamente.

Una skill no es sólo un prompt más largo. Es una estructura modular que encapsula todo lo que un agente necesita para realizar un dominio específico de tareas. Cuando el agente detecta que una tarea coincide con una skill disponible, carga esa skill y la usa como marco de referencia.

2. Skills vs. Prompts: La diferencia clave

Es fácil confundir skills con prompts, pero son conceptos distintos con propósitos diferentes:

Prompt simple

  • Estático: Se define una vez y no cambia
  • Genérico: Funciona para muchas tareas del mismo tipo
  • Sin estado: No recuerda interacciónes anteriores
  • Sin herramientas: Sólo genera texto
  • Sin composición: No se combina con otros prompts

Skill

  • Dinámico: Se carga y descarga bajo demanda
  • Especializado: Optimizado para un dominio específico
  • Con estado: Puede mantener contexto entre pasos
  • Con herramientas: Integra funciones ejecutables
  • Componible: Se combina con otras skills

Ejemplo práctico: Un prompt diría "Sos un asistente de atención al cliente". Una skill de atención al cliente incluiría: las instrucciónes de tono y protocolo, acceso a la base de conocimiento de productos, funciones para consultar el estado de pedidos, plantillas de respuestas para problemas comunes, y reglas de escalamiento a un humano.

3. Composición de skills

Una de las ventajas más poderosas de las skills es que se pueden combinar. Un agente puede cargar múltiples skills simultáneamente para resolver tareas que requieren conocimiento de varios dominios:

Skill: Análisis de datos
+
Skill: Visualización
+
Skill: Escritura técnica
=
Agente que genera informes con gráficos

La composición permite crear agentes versátiles sin necesidad de entrenar un modelo gigantesco para todo. En lugar de eso, se arma un "set de habilidades" según la tarea. Es como darle a un empleado diferentes herramientas según el proyecto: para uno necesita un destornillador, para otro una sierra, y para otro ambas.

4. Carga dinámica de skills

El proceso de carga dinámica permite al agente seleccionar y activar skills en tiempo de ejecución, sin necesidad de reiniciar o reconfigurar el modelo:

1
El usuario envía una tarea: "Genera un informe de ventas del Q1 con gráficos"
2
El enrutador analiza la intención y detecta que necesita: análisis de datos + visualización + escritura
3
Se cargan las skills relevantes al contexto: skill_analisis, skill_graficos, skill_informes
4
El agente ejecuta la tarea usando las instrucciónes y herramientas de cada skill activa
5
Al completar, las skills se descargan del contexto para liberar espacio de tokens

5. Plataformas y su implementación de skills

Diversas plataformas han implementado variantes del concepto de skills para sus agentes:

OpenAI (Custom GPTs)

Permite crear "Custom GPTs" con instrucciónes, archivos y acciónes (APIs). Cada Custom GPT es esencialmente una skill empaquetada que se puede compartir y reutilizar.

Anthropic (Claude Projects)

Projects permite definir instrucciónes de sistema, archivos de referencia y personalización. Funciona como un workspace con skills pre-cargadas para un dominio específico.

LangChain (Tools + Agents)

Tools en LangChain son equivalentes a skills atómicas. Se combinan en agents que pueden usar múltiples tools según la necesidad de cada paso.

OpenAI (Assistants API)

Los Assistants permiten definir "instructions" (skill base) + "tools" (herramientas) + "file_search" (contexto). Es la implementación más cercana a una skill completa.

6. Simulación: Sistema de skills en acción

Veamos una implementación simplificada de cómo un sistema de skills funciona en la práctica:

class Skill:
    def __init__(self, nombre, instrucciones, herramientas=None, contexto=None):
        self.nombre = nombre
        self.instrucciones = instrucciones   # Prompt especializado
        self.herramientas = herramientas or []  # Funciones disponibles
        self.contexto = contexto or ""        # Datos de referencia

class AgenteConSkills:
    def __init__(self, llm, biblioteca_skills):
        self.llm = llm
        self.biblioteca = biblioteca_skills  # Dict de skills disponibles
        self.skills_activas = []
    
    def cargar_skill(self, nombre_skill):
        """Carga una skill al contexto activo del agente"""
        if nombre_skill in self.biblioteca:
            skill = self.biblioteca[nombre_skill]
            self.skills_activas.append(skill)
            print(f"  Skill cargada: {skill.nombre}")
    
    def resolver_tarea(self, tarea):
        # 1. Analizar qué skills necesita la tarea
        skills_necesarias = self.analizar_necesidades(tarea)
        
        # 2. Cargar las skills relevantes
        for nombre in skills_necesarias:
            self.cargar_skill(nombre)
        
        # 3. Construir el contexto combinado
        contexto_total = self.construir_contexto()
        
        # 4. Ejecutar con el LLM usando las skills activas
        return self.llm.predict(f"""
            {contexto_total}
            
            Tarea: {tarea}
            
            Resuelve la tarea usando las instrucciones
            y herramientas de las skills cargadas.
        """)
    
    def construir_contexto(self):
        """Combina instrucciones y contexto de todas las skills activas"""
        partes = []
        for skill in self.skills_activas:
            partes.append(f"--- Skill: {skill.nombre} ---")
            partes.append(skill.instrucciones)
            if skill.contexto:
                partes.append(f"Contexto: {skill.contexto}")
            if skill.herramientas:
                nombres = [h.__name__ for h in skill.herramientas]
                partes.append(f"Herramientas: {', '.join(nombres)}")
        return "\n".join(partes)

# Ejemplo de uso
biblioteca = {
    "analisis_datos": Skill(
        nombre="Análisis de datos",
        instrucciones="Analiza datasets y genera estadísticas descriptivas.",
        herramientas=[calcular_promedio, generar_estadisticas],
        contexto="Usa métricas estándar: media, mediana, desviación."
    ),
    "visualizacion": Skill(
        nombre="Visualización",
        instrucciones="Crea gráficos claros y informativos.",
        herramientas=[crear_grafico_barras, crear_grafico_lineas],
        contexto="Formato: SVG, colores accesibles, leyendas claras."
    )
}

agente = AgenteConSkills(llm=modelo, biblioteca_skills=biblioteca)
resultado = agente.resolver_tarea(
    "Genera un informe de ventas del Q1 con gráficos de tendencia"
)

El poder de las skills: En lugar de tener un prompt gigantesco que intente cubrir todas las posibles tareas, las skills permiten un enfoque modular. El agente carga sólo lo que necesita, lo que reduce el consumo de tokens, mejora la precisión de las respuestas, y permite que la biblioteca de skills crezca sin degradar el rendimiento.

5.4 MCP: Model Context Protocol

Imagina que cada dispositivo electrónico tuviera su propio tipo de puerto de carga: uno para cables redondos, otro para cables planos, otro magnético. Sería un caos. USB unificó todo eso: un sólo puerto que funciona con cualquier dispositivo. El Model Context Protocol (MCP), creado por Anthropic, hace lo mismo para los agentes de IA: un estándar universal que permite conectar cualquier modelo con cualquier fuente de datos o herramienta, sin importar quién hizo cada uno.

La analogía USB: Antes de MCP, cada integración entre un LLM y una herramienta era custom: si querías que tu modelo accediera a Notion, escribías código específico para Notion. Si luego querías Slack, escribías otro código diferente. MCP estandariza esa conexión: el proveedor de datos implementa el protocolo una vez, y cualquier modelo compatible puede conectarse automáticamente. Es un "puerto universal" para la IA.

1. ¿Qué es MCP exactamente?

MCP es un protocolo de comunicación abierto que define cómo los modelos de IA (clientes) se conectan con fuentes de datos y herramientas externas (servidores). No es una API, no es un SDK, no es un framework: es un protocolo, como HTTP lo es para la web. Cualquier plataforma puede implementarlo.

Estándar abierto

Cualquier desarrollador puede implementar un servidor MCP o un cliente MCP. No hay vendor lock-in ni licencias restrictivas.

Transporte flexible

Funciona sobre stdio (para procesos locales), HTTP/SSE (para servicios remotos) y potencialmente WebSocket. Se adapta al entorno de ejecución.

Descubrimiento automático

El cliente puede preguntar al servidor qué herramientas, recursos y prompts tiene disponibles, sin configuración previa.

Seguridad integrada

El protocolo define mecanismos de autenticación y control de acceso para proteger los datos del servidor.

2. Arquitectura cliente-servidor

MCP utiliza una arquitectura cliente-servidor donde el cliente es la aplicación de IA (como Claude Desktop o un agente custom) y el servidor es la fuente de datos o herramienta que se conecta:

Cliente MCP

La aplicación de IA que necesita acceder a datos y herramientas. Ej: Claude Desktop, VS Code con Copilot, un agente personalizado.

⇄

Servidor MCP

El proveedor de datos o herramientas. Ej: un servidor de Notion, de GitHub, de una base de datos, de un sistema de archivos.

La comunicación entre cliente y servidor se realiza mediante mensajes JSON-RPC 2.0. El cliente envía solicitudes y el servidor responde con resultados. También puede enviar notificaciones unidireccionales para actualizaciones en tiempo real.

3. Los tres pilares: Recursos, Herramientas y Prompts

MCP define tres tipos de primitives que un servidor puede exponer. Cada una tiene un propósito distinto y se usa de manera diferente:

Recursos (Resources)

Datos que el servidor pone a disposición del modelo. Son sólo lectura. Ej: el contenido de un archivo, datos de una API, registros de una base de datos. El modelo puede leerlos pero no modificarlos directamente.

Herramientas (Tools)

Funciones que el modelo puede invocar para ejecutar acciónes. Son ejecutables. Ej: crear un archivo, enviar un mensaje, ejecutar una consulta SQL. Requieren aprobación del usuario en muchos casos.

Prompts

Plantillas predefinidas que el servidor puede ofrecer al modelo. Ayudan a estructurar interacciónes comunes. Ej: un prompt para revisar código, otro para resumir documentos, otro para generar tests.

La diferencia clave: Las recursos son datos pasivos (como abrir un libro). Las herramientas son acciónes activas (como usar un destornillador). Los prompts son plantillas de interacción (como un formulario pre-armado). Un servidor MCP puede exponer las tres cosas o sólo una, dependiendo de su diseño.

4. ¿Por qué importa la estandarización?

Sin un protocolo estándar, cada integración entre LLMs y herramientas es un proyecto custom. Esto crea un problema de escalabilidad:

Sin MCP (custom) Con MCP (estándar)
N proveedores × M modelos = N×M integraciones N proveedores + M modelos = N+M implementaciones
Cada integración es única y frágil Una implementación funciona con todos los clientes
Actualizar una API rompe la integración El protocolo gestiona compatibilidad
No hay descubrimiento automático El cliente descubre herramientas al conectarse
Cada desarrollador resuelve autenticación a su manera Mecanismos de auth estandarizados

La estandarización también beneficia a los desarrolladores de herramientas: implementan el protocolo una vez y su herramienta funciona con Claude, GPT, Gemini, y cualquier otro modelo compatible. No necesitan escribir integración separada para cada proveedor de IA.

5. MCP vs. Function Calling

MCP y function calling resuelven problemas similares pero en niveles diferentes. No son competidores; son complementarios:

Aspecto Function Calling MCP
Alcance Llamadas a funciones específicas definidas por el desarrollador Protocolo completo para conectar fuentes de datos y herramientas
Descubrimiento Las funciones se definen estáticamente en el código El cliente descubre automáticamente qué está disponible
Portabilidad Cada proveedor tiene su propio formato Protocolo universal compatible con múltiples proveedores
Recursos Sólo ejecuta funciones También accede a datos (recursos) y ofrece prompts
Transporte HTTP directo o SDK del proveedor stdio, HTTP/SSE,WebSocket con handshake definido

En la práctica, un agente puede usar MCP para conectarse a un servidor que, internamente, expone sus capacidades como tools. MCP actúa como la capa de transport y descubrimiento, mientras que las tools individuales son las unidades de ejecución.

6. Implementaciones reales y el ecosistema

Desde su lanzamiento a finales de 2024, MCP ha generado un ecosistema creciente de implementaciones:

Claude Desktop

Fue la primera aplicación en soportar MCP. Permite conectar servidores locales y remotos para que Claude acceda a archivos, bases de datos y APIs.

Servidores comunitarios

Existen servidores MCP para: GitHub, Notion, Slack, PostgreSQL, MySQL, Slack, Google Drive, Filesystem, Puppeteer, y muchos más.

SDK oficial

Anthropic provee SDKs en TypeScript y Python para construir servidores y clientes MCP. Simplifican la implementación del protocolo.

Adopción

OpenAI, Google y otros proveedores han anunciado compatibilidad o interés en MCP, convirtiéndolo en un estándar de facto emergente.

7. Simulación: Un servidor MCP en acción

Veamos cómo se ve la interacción entre un cliente MCP y un servidor que expone acceso a una base de datos:

import json

class ServidorMCP:
    """Simulación de un servidor MCP que expone una base de datos"""
    
    def __init__(self):
        self.herramientas = {
            "consultar_usuarios": {
                "descripcion": "Busca usuarios por nombre o email",
                "parametros": {
                    "tipo": "object",
                    "propiedades": {
                        "nombre": {"tipo": "string"},
                        "email": {"tipo": "string"}
                    }
                }
            },
            "crear_pedido": {
                "descripcion": "Crea un nuevo pedido para un usuario",
                "parametros": {
                    "tipo": "object",
                    "propiedades": {
                        "usuario_id": {"tipo": "integer"},
                        "productos": {"tipo": "array"},
                        "direccion": {"tipo": "string"}
                    },
                    "requeridos": ["usuario_id", "productos"]
                }
            }
        }
        
        self.recursos = {
            "file:///docs/catalogo.json": {
                "nombre": "Catálogo de productos",
                "tipo": "application/json"
            },
            "file:///docs/politicas.md": {
                "nombre": "Políticas de envío",
                "tipo": "text/markdown"
            }
        }
    
    def manejar_solicitud(self, solicitud):
        """Procesa mensajes JSON-RPC 2.0 del cliente"""
        metodo = solicitud.get("method")
        params = solicitud.get("params", {})
        
        if metodo == "tools/list":
            # El cliente pregunta: ¿qué herramientas tenés?
            return {
                "tools": [
                    {"name": k, "description": v["descripcion"],
                     "inputSchema": v["parametros"]}
                    for k, v in self.herramientas.items()
                ]
            }
        
        elif metodo == "tools/call":
            # El cliente pide ejecutar una herramienta
            nombre = params.get("name")
            args = params.get("arguments", {})
            return self.ejecutar_herramienta(nombre, args)
        
        elif metodo == "resources/list":
            # El cliente pregunta: ¿qué recursos tenés?
            return {
                "resources": [
                    {"uri": uri, "name": r["nombre"], "mimeType": r["tipo"]}
                    for uri, r in self.recursos.items()
                ]
            }
        
        elif metodo == "resources/read":
            # El cliente quiere leer un recurso específico
            uri = params.get("uri")
            return self.leer_recurso(uri)

# Flujo de la conversación cliente ↔ servidor
print("1. Cliente se conecta al servidor MCP")
print("2. Cliente envía: tools/list → Descubre las herramientas disponibles")
print("3. Cliente envía: resources/list → Descubre los recursos disponibles")
print("4. LLM decide usar 'consultar_usuarios' con nombre='Pablo'")
print("5. Cliente envía: tools/call con los argumentos")
print("6. Servidor ejecuta la consulta y retorna resultados")
print("7. LLM integra los resultados en una respuesta natural")

Esto reduce drásticamente la fricción para crear agentes verdaderamente útiles que puedan interactuar con el mundo real sin necesidad de integraciones custom para cada servicio.
