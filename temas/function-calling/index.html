Function Calling: Uso de herramientas

Un LLM por sí solo solo puede generar texto. Function calling es el mecanismo que le permite decir "necesito usar esta herramienta con estos datos" en lugar de simplemente inventar una respuesta. Es la diferencia entre un modelo que te dice "hace calor hoy" y uno que consulta una API del clima y te da la temperatura exacta de tu ciudad.

La analogía del restaurante: Piensa en un LLM como un cliente en un restaurante. El cliente sabe qué quiere comer (el objetivo), pero no puede cocinar. Le dice al mesero (el sistema): "Quiero una pasta al pesto sin frutos secos". El mesero traduce eso a una orden específica (function call) para la cocina (API externa), la cocina prepara el plato, y el mesero regresa con el resultado (la respuesta de la función). El cliente nunca entra a la cocina — solo recibe el resultado.

1. ¿Cómo funciona el mecanismo?

El proceso de function calling sigue un flujo preciso entre el usuario, el LLM y las herramientas externas:

👤
Usuario "¿Qué tiempo hace en Buenos Aires mañana?"
↓
🧠
LLM analiza Identifica que necesita datos externos. Decide llamar a obtener_clima(ciudad="Buenos Aires", fecha="mañana")
↓
⚡
Sistema ejecuta Envía la llamada a la API del clima con los parámetros exactos
↓
📡
API responde Retorna: {"temp": 22, "condicion": "parcialmente nublado", "lluvia": 30}
↓
🧠
LLM integra "Mañana en Buenos Aires habrá 22°C con cielo parcialmente nublado. Hay 30% de probabilidad de lluvia, llevá paraguas."

Punto clave: El LLM nunca ejecuta la función directamente. Solo genera la llamada con el nombre correcto y los parámetros correctos. Es el sistema externo (el runtime del agente) quien realmente conecta con la API y ejecuta la operación.

2. Formato de la llamada: Request y Response

Cada function call tiene un formato estructurado. Primero, el desarrollador define las herramientas disponibles para el modelo. Luego, cuando el modelo decide usar una herramienta, genera un JSON con la estructura esperada:

Paso 1: Definición de herramientas 
{
  "nombre": "obtener_clima",
  "descripcion": "Obtiene el pronóstico del clima para una ciudad",
  "parametros": {
    "type": "object",
    "properties": {
      "ciudad": {
        "type": "string",
        "description": "Nombre de la ciudad"
      },
      "fecha": {
        "type": "string",
        "enum": ["hoy", "mañana", "pasado"],
        "description": "Para qué día se consulta"
      }
    },
    "required": ["ciudad"]
  }
}
Paso 2: Respuesta del modelo (function call) 
{
  "tool_calls": [
    {
      "id": "call_abc123",
      "function": {
        "name": "obtener_clima",
        "arguments": "{\"ciudad\": \"Buenos Aires\", \"fecha\": \"mañana\"}"
      }
    }
  ]
}
Paso 3: Resultado devuelto al modelo 
{
  "tool_call_id": "call_abc123",
  "output": {
    "ciudad": "Buenos Aires",
    "temperatura": 22,
    "condicion": "parcialmente nublado",
    "probabilidad_lluvia": 30,
    "viento_kmh": 15
  }
}

3. Validación de parámetros

El sistema valida que los parámetros generados por el modelo coincidan con el esquema definido. Esto previene errores en tiempo de ejecución:

Parámetro Tipo Requerido Validación
ciudad string Sí No puede estar vacío
fecha enum No Debe ser "hoy", "mañana" o "pasado"
unidades enum No Debe ser "celsius" o "fahrenheit"

Si el modelo genera un parámetro inválido (por ejemplo, una fecha fuera del enum), el sistema puede rechazar la llamada antes de ejecutarla, o devolver un error que el modelo puede usar para corregir su respuesta.

4. Llamadas encadenadas y ejecución paralela

Las tareas complejas suelen requerir múltiples llamadas a herramientas. Hay dos patrones principales:

Encadenamiento secuencial: Cada llamada depende del resultado de la anterior. El modelo genera una llamada, recibe el resultado, decide la siguiente, y así sucesivamente.

buscar_usuario("pablo")
→
obtener_pedidos(id=42)
→
calcular_total(pedidos)

Ejecución paralela: Cuando las llamadas son independientes, el modelo puede generar múltiples tool calls en una sola respuesta. El sistema las ejecuta simultáneamente y devuelve todos los resultados juntos.

obtener_clima("Buenos Aires")
+
obtener_clima("Córdoba")
+
obtener_clima("Mendoza")

5. Ejemplo real: Consulta a una base de datos

Veamos un ejemplo completo de cómo un asistente de e-commerce usa function calling para responder preguntas sobre pedidos:

import json

# Herramientas disponibles para el asistente
herramientas = [
    {
        "nombre": "buscar_producto",
        "descripcion": "Busca productos por nombre o categoría",
        "parametros": {
            "consulta": "string (requerido)",
            "categoria": "string (opcional)",
            "precio_maximo": "number (opcional)"
        }
    },
    {
        "nombre": "consultar_stock",
        "descripcion": "Verifica disponibilidad de un producto",
        "parametros": {
            "producto_id": "string (requerido)",
            "sucursal": "string (opcional)"
        }
    },
    {
        "nombre": "crear_pedido",
        "descripcion": "Crea un nuevo pedido con los productos indicados",
        "parametros": {
            "productos": "array de {id, cantidad} (requerido)",
            "direccion_envio": "string (requerido)"
        }
    }
]

# Simulación: el usuario hace una pregunta
pregunta = "¿Tienen auriculares bluetooth por menos de $50.000?"

# El LLM decide qué herramienta usar
decision_llm = {
    "tool_calls": [{
        "function": {
            "name": "buscar_producto",
            "arguments": json.dumps({
                "consulta": "auriculares bluetooth",
                "precio_maximo": 50000
            })
        }
    }]
}

# El sistema ejecuta la función
resultado_real = buscar_producto(
    consulta="auriculares bluetooth",
    precio_maximo=50000
)
# → [{"id": "AUR001", "nombre": "Sony WH-CH520", "precio": 35000},
#    {"id": "AUR002", "nombre": "JBL Tune 510BT", "precio": 28000}]

# El LLM integra el resultado en una respuesta natural
respuesta = "Sí, tenemos dos opciones por debajo de $50.000:
  1. Sony WH-CH520 — $35.000
  2. JBL Tune 510BT — $28.000
  ¿Querés que verifique el stock de alguno?"

La ventaja de function calling: El LLM no "inventa" precios ni inventarios. Consulta datos reales del sistema y genera una respuesta precisa y actualizada. Si no encuentra resultados, simplemente lo informa en lugar de fabricar información.

6. Consideraciones importantes

Function calling no es infalible. Hay varios aspectos a tener en cuenta al diseñar un sistema con herramientas:

  • El modelo puede equivocarse: A veces genera parámetros incorrectos o elige la herramienta equivocada. Siempre validar las llamadas antes de ejecutarlas.
  • Seguridad: Si una función puede modificar datos o ejecutar código, el agente necesita permisos bien definidos. Nunca expongas funciones destructivas sin validación humana.
  • Latencia: Cada llamada a herramienta agrega tiempo. Las llamadas paralelas mitigan esto, pero el encadenamiento secuencial puede ser lento.
  • Límites de la ventana: El resultado de cada función consume tokens en la ventana de contexto. Si los resultados son muy grandes, el modelo puede perder información importante.
